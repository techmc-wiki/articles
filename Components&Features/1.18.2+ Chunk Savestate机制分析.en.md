## TL;DR

`TACS` (`ThreadedAnvilChunkStorage`) saves chunk A (old state) during incremental save in a normal tick and sets a 10-second cooldown →
States of chunks A and B are changed →
In subsequent save attempts, chunk A is skipped due to cooldown, chunk B is saved normally →
Main thread freezes, Watchdog force-exits →
After restart, chunk A is the old snapshot, chunk B is the new snapshot, items exist in both locations simultaneously.

## TACS

`TACS` (`ThreadedAnvilChunkStorage`), corresponding to the `ChunkMap` class in Mojang mappings, is the core component for server-side chunk persistence.
It is responsible for deciding which chunks need to be saved, when to save them, and throttling control for saves.

`TACS` holds references to `ChunkHolder` (chunk holder, the wrapper object for each loaded chunk in `TACS`) for all loaded chunks.
These references are stored in `visibleChunkMap` (a `Long2ObjectLinkedOpenHashMap<ChunkHolder>`, with chunk coordinate long encoding as key).
`TACS` drives the serialization and write pipeline for chunks on every tick and during auto-saves.

All save logic mentioned below (incremental saves, auto-saves, cooldown throttling) is completed within `TACS`.

## Actual Triggering of Chunk Saves

Chunk saves don't only happen during the auto-save that occurs once every 6000 ticks.
Every tick, the server performs incremental saves.

The call chain is as follows:

```
MinecraftServer.tickServer()
  → tickChildren()
    → serverLevel.tick()
      → chunkSource.tick()            // ServerChunkCache.tick()
        → chunkMap.tick()             // ChunkMap.tick()
          → processUnloads()
            → Iterate visibleChunkMap, save up to 20 chunks per tick
```

At the end of `ChunkMap.processUnloads()` (1.19.2 `ChunkMap.java`:502-509):

```java
int k = 0;
ObjectIterator<ChunkHolder> objectIterator = this.visibleChunkMap.values().iterator();
while (k < 20 && booleanSupplier.getAsBoolean() && objectIterator.hasNext()) {
    if (this.saveChunkIfNeeded(objectIterator.next())) {
        k++;
    }
}
```

Up to 20 chunks are attempted to be saved per tick. This loop is independent of auto-save and runs continuously in the background.

## 10-Second Save Cooldown

The 10-second save cooldown is `TACS`'s throttling mechanism for individual chunk save frequency.
Its purpose is to avoid frequently changing chunks (such as chunks with continuous redstone activity) being serialized and written to disk every tick, thereby reducing I/O overhead.

The specific implementation is in `ChunkMap.saveChunkIfNeeded()` (1.19.2 `ChunkMap.java`:762-786):

```java
long m = this.chunkSaveCooldowns.getOrDefault(l, -1L);
long n = System.currentTimeMillis();
if (n < m) {
    return false;   // Within cooldown period, skip save
}
boolean bl = this.save(chunkAccess);
if (bl) {
    this.chunkSaveCooldowns.put(l, n + 10000L);  // Set 10-second cooldown after successful save
}
```

`chunkSaveCooldowns` is a `Long2LongOpenHashMap`, with key being the long encoding of chunk position, and value being the timestamp (ms) when the next save is allowed.
After a chunk is successfully saved, its cooldown deadline is set to current time + 10000ms.

During the cooldown period, even if the chunk has become dirty again due to block changes (`needsSaving == true`, i.e., the chunk's dirty flag is set), `saveChunkIfNeeded()` will directly return `false` and skip the save.
The dirty flag is only a necessary condition for saving, not a sufficient condition. When the cooldown hasn't expired, dirty chunks are still skipped.

## Serialization Completes Synchronously on Main Thread

Key sequence in `ChunkMap.save()` (1.19.2 `ChunkMap.java`:788-818):

```java
if (!chunkAccess.isUnsaved()) { return false; }
chunkAccess.setUnsaved(false);                                          // Clear dirty flag
CompoundTag compoundTag = ChunkSerializer.write(this.level, chunkAccess); // Main thread synchronous serialization
this.write(chunkPos, compoundTag);                                       // Submit to IOWorker (background worker thread responsible for async disk writes)
```

`ChunkSerializer.write()` executes inline on the main thread, iterating through `chunk.getBlockEntityPositions()`,
calling `chunk.getPackedBlockEntityNbt()` for each position.
The serializer sees the logical state of the live chunk (the active chunk object in memory being used by game logic) at the moment of invocation.
There is no scenario where the I/O thread reads from the active chunk memory.

## Async Write Pipeline — Three-Layer Chain

The serialized NBT goes through three layers before reaching the disk:

**Layer 1**: Main thread → `IOWorker.pendingWrites` (JVM heap memory)

`ChunkStorage.write()` → `IOWorker.store()`, stores NBT into `pendingWrites`, a `LinkedHashMap`.

**Layer 2**: `IOWorker` background thread → `RegionFile` (OS file cache)

`IOWorker.storePendingChunk()` takes one entry at a time, calls `RegionFile.write()`.
`RegionFile.write()` calls `FileChannel.write()` to write data to the file, but does not call `force()`/`fsync()` (OS-level forced flush system call).
Data enters the OS's page cache (the area in memory where the OS caches file writes; after writing, data is temporarily stored here rather than directly written to physical disk), not yet guaranteed to reach physical disk.

**Layer 3**: OS page cache → Physical disk

Without `fsync`, flushing to disk depends on the OS's dirty page writeback strategy.
When the process terminates, unflushed data may be lost.

`IOWorker` writes chunks entry by entry, not as an atomic commit of the entire world at once.

## Auto-Save Does Not Bypass Cooldown

Auto-save triggered every 6000 ticks (`MinecraftServer.tickServer()` 1.19.2:828):

```java
if (this.tickCount % 6000 == 0) {
    this.saveEverything(true, false, false);  // flush=false
}
```

The `flush` parameter here determines the thoroughness of save behavior:
- `flush=true`: Forces serialization of all dirty chunks (bypasses 10-second cooldown), and waits for `IOWorker` to finish writing all queued data to disk before returning. Used for `/save-all flush` or server shutdown.
- `flush=false`: Goes through the normal `saveChunkIfNeeded()` path, subject to cooldown restrictions, and returns immediately after submitting to `IOWorker`, without waiting for actual disk completion.

Auto-save uses `flush=false`.

`saveEverything` → `saveAllChunks` → `serverLevel.save()` → `serverChunkCache.save(false)` → `ChunkMap.saveAllChunks(false)`.

The non-flush branch of `saveAllChunks(false)` (1.19.2 `ChunkMap.java`:452-454):

```java
this.visibleChunkMap.values().stream()
    .filter(ChunkHolder::wasAccessibleSinceLastSave)
    .forEach(this::saveChunkIfNeeded);    // Reuses saveChunkIfNeeded, subject to cooldown restrictions
```

Auto-save calls the same `saveChunkIfNeeded()`.
If a chunk was just saved in a previous incremental save, auto-save will also skip it during the cooldown period.

## When Chunks Become Dirty

When `WorldChunk.setBlockState()` executes on the main thread (1.19.2):
1. Modify block state inside section
2. Call `onStateReplaced` on old state
3. `removeBlockEntity` if necessary
4. Finally set `needsSaving = true`

A chunk becomes dirty again as soon as a block change occurs.
However, the dirty state has no writeback relationship with the old NBT already queued in `IOWorker` — the old NBT is already an independent data object.

## Watchdog Force Exit

`DedicatedServerWatchdog` monitors the main thread in a separate thread.
After detecting timeout:
1. Write crash report
2. Call `System.exit(1)`
3. Schedule `Runtime.halt(1)` as fallback after 10 seconds

`System.exit(1)` triggers the JVM shutdown process.
Before complete exit, the `IOWorker` background thread may still continue running for a short time, continuing to write out chunks already queued in `pendingWrites`.
Until the JVM exits normally or is hard-killed by `halt(1)`.

## Process Walkthrough

Using the dispenser shooting a shulker box to an adjacent chunk as an example, as demonstrated in Comedy's video:

**T=0: Incremental save saves chunk A**

During a normal tick, the incremental save loop iterates to chunk A (containing dispenser + shulker box), conditions satisfied:
- Cooldown has expired
- `needsSaving == true`

Main thread synchronously executes `ChunkSerializer.write()`, generating `A_old_nbt` (dispenser contains shulker box).
Submitted to `IOWorker`.
Set `chunkSaveCooldowns[A] = now + 10000ms`.

Chunk A then enters a 10-second cooldown period.

**T=1: Chunks A and B are changed**

Dispenser is activated by redstone, 4 ticks later `ServerTickList.tick()` executes the scheduled tick, dispenser triggers.
Shulker box moves from chunk A (dispenser) to chunk B (landing position).
Chunk A becomes dirty (dispenser becomes empty), chunk B becomes dirty (shulker box block appears).

**T=2: Auto-save/incremental save runs again**

Auto-save triggers, or incremental save loop iterates to these two chunks:
- Chunk A: `saveChunkIfNeeded()` → cooldown not expired (still within 10 seconds) → **skipped**
- Chunk B: No cooldown or cooldown expired → save successful → generates `B_new_nbt` (contains shulker box)

At this point the persistent state has diverged:
- Chunk A: On disk is `A_old_nbt` (dispenser contains shulker box)
- Chunk B: On disk is `B_new_nbt` (landing position has shulker box)

**T=3: Main thread frozen by redstone + Watchdog force-exit**

After main thread freezes, no new serialization occurs.
`IOWorker` background thread continues writing remaining NBT in `pendingWrites`.
Watchdog detects timeout, `System.exit(1)`.

Note: The state fork was already formed in the T=2 phase — chunk A was skipped due to cooldown, chunk B was already saved as a new snapshot.
The Watchdog force-exit at T=3 only prevents subsequent save opportunities that might have corrected this fork (such as the next incremental save after cooldown expiration).
Additionally, since `RegionFile.write()` does not call `fsync` (see Section 5), data in the OS page cache that hasn't been flushed to physical disk when the JVM exits may also be lost, but this is only an additional factor to the fork, not the core cause.

**After Restart**

Chunk A loads from disk: dispenser contains shulker box (old snapshot).
Chunk B loads from disk: landing position has shulker box (new snapshot).
Shulker box is duplicated.
