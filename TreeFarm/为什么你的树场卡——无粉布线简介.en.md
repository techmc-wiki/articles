Many of you have probably been waiting for this for a while. Dustless tree farms have seemed like a pretty good way to "show off" in recent years. However, Xinghe has always emphasized:

> Dustless has never been the goal, but rather a means. The ultimate objective should be to reduce lag, not to force dustless designs. Forcing dustless designs can easily lead to increased lag, which runs counter to the fundamental philosophy of dustless wiring.

Understanding this point allows us to truly make good use of dustless wiring.

Next, we'll explain the main methods of dustless wiring step by step.

## Rails and Observers

As we all know, rails **emit PP updates simultaneously** when their activation state changes, and can be detected by Observers. Rails and Observers form the most basic dustless signal transmission method.

![Observer_Railway](./img/Observer_Railway.png)

In tree farms, Pistons are generally arranged vertically. We only need to activate a vertical column of Observers simultaneously to activate an entire row of Pistons at once. There are two basic methods: by pushing **the Observer itself** or **the block the Observer is facing** with a Piston. ~~Resetting this row of things can be done casually~~

![Observer_Column](./img/Observer_column.png)

![observer_column2](./img/observer_column2.png)

Since Observers have a built-in 2gt delay, we can use them to construct macro-level timing. There are already many examples in this direction, so we won't analyze them in detail.

You can also use BUD rails for zero-delay signal transmission:

![BUD_Railway](./img/BUD_Railway.png)

## Tree Electricity/Scaffold Electricity/Wall Electricity

_Actually there's also water electricity, but since nobody uses it, we'll just ignore it here_

### Tree Electricity

As we all know, starting from 1.14, after receiving a PP update, leaves will schedule a tick (if in TT, it executes immediately; otherwise it executes **in TT 1gt later**) to check their distance from logs. If the distance changes, they emit **NC and PP updates**. In tree farms, we generally push or pull logs with Pistons or Slime Blocks to achieve distance changes.

> In fact, tree electricity's update mechanism is somewhat like Redstone Dust: it checks the distance of surrounding logs and leaves. If there's an adjacent log, it sets its own distance to 0; if there are no logs around, it sets its own distance to the minimum distance of adjacent leaves + 1.

Through tree electricity, we can create many useful **odd gt delay** devices, such as making an upward-pushing column start pushing down at 3gt.

![leaves_down](./img/leaves_down.png)

> Tree electricity detection utilizes the characteristic that tree electricity has zero delay within TT. The tree farm's growth module **runs within TT**, so trees inevitably grow and place logs within TT. Therefore, leaves will immediately execute the check within TT, then emit NC and PP updates, triggering the BUD and achieving growth detection. However, due to space constraints, this is almost no longer used.
>
> ![Leave_Detection2](./img/Leave_Detection_root.png)

### Scaffold Electricity

As we all know, when a scaffold receives a PP update, it schedules a tick to check its **horizontal distance** from the supporting block. If the distance changes (and is not greater than 6; greater than 6 will cause it to drop), it emits NC and PP updates. Scaffold electricity adds **1gt delay at each scaffold**. Generally, we trigger scaffold electricity through **trapdoor state changes** below the scaffold.

![scafolding](./img/scafolding_upd.png)

Generally, we use scaffold electricity to create **odd gt delays** ~~it looks similar to tree electricity, and the use cases are almost the same~~

### Wall Electricity

As we all know, starting from 1.16, if there are connecting blocks on both the x and z axes, the wall is low; otherwise it's high. If there's one high wall in an entire column of walls, then starting from it, all walls below it become high walls (low walls don't override). We generally achieve wall state changes through **opening/closing fence gates above** the topmost wall or **opening/closing doors/trapdoors on the side**.

![wall_upd1](./img/Wall_upd.png)

![wall_upd2](./img/Wall_upd2.png)

Additionally, walls actively connect to **certain blocks above them**. Whether connected or not is also a state of the wall, which can be used to redirect block streams upward.

![wall_blockstream](./img/Wall_blockstream.png)

Walls are **instant components**. All wall state changes emit **PP updates** and are detected by Observers.

> Designs that push the connecting blocks or even the walls themselves also exist, especially for single-edge wall electricity. For more detailed content, you can refer to [Dongdong's video](https://www.bilibili.com/video/BV1Zy4y117DC)

## Signal Transmission Without Macro Delay

The signal transmission methods mentioned above all have **macro delay**. If you tried wiring after the previous chapter, you should have found that **we really need signal transmission methods without macro delay**. Here, we'll briefly understand the two most important signal transmission methods without macro delay in dustless wiring.

### Slime Sticks

![slime](./img/Slime_stick.png)

Slime sticks, as the name suggests, use Slime Blocks to pull things to complete signal transmission. As we all know, when a Piston pushes or pulls a chain of blocks, **it deletes the blocks at their original positions during the BE phase**, so **it can change another Piston's activation state within the same gt**, thus eliminating macro delay.

For sticky Pistons in such chains, **falling edge transmission has no macro delay, while rising edges have 3gt delay per Piston**. What to do? Generally, we add an auto-reset device to each stick that automatically resets at 3gt, allowing them to reset synchronously and **output rising edge signals synchronously**.

### Rails + BUD

![BUD+Railway](./img/BUD+Railway.png)

Remember that rails are instant components? By using rails to update BUDs, and having the BUD change the activation state of another chain of rails, we can keep the signal transmitting within the BE phase.

Such structures generally also have zero-delay falling edges (_because in a dustless environment, our only wiring method that can output rising edges with zero delay is relatively complex_) and auto-reset to output rising edges.

> In fact, for BUDs, they only need updates, regardless of rising or falling edge signals. This description is just to maintain consistency with the previous text.

## Redstone Redirection

_Since its English name is Redstone Redirection, we don't use the common translation "转向"_

This is the **only method in a dustless environment that can output rising edge signals with zero delay** mentioned earlier.

As we all know, Redstone Dust **only powers in the direction it points**, so as long as we create a **powered Redstone Dust that doesn't point in a certain direction**, and when inputting a rising edge signal, **make it redirect to point in a certain direction**, we can output a rising edge signal in that direction with zero delay.

![Redirect](./img/Redstone_redirect_power.png)

As we all know, Redstone Dust is redirected by the following blocks:

- Repeaters, Comparators, tripwire hooks _immovable_
- **Redstone blocks, Observers, targets, lightning rods, detector rails** _movable_

Among them, **the movable ones** are our most important practical methods for using redstone redirection.

Note that redstone redirection **only includes PP updates**. To activate certain components, we need to **additionally provide NC updates**.

Below we'll briefly introduce several components based on redstone redirection that are **heavily used** in tree farms.

### Columns

Very simple - we arrange several redstone redirections vertically. You can naturally think of arranging the blocks used for redirection vertically and moving them vertically.

![pistonwall](./img/Redstone_Redirect_Pistonwall.png)

This way we easily obtain an entire column of **synchronized** redstone redirections, making it easy to activate an entire row of Pistons.

### 3gt Generator

This is a standard 3gt generator.

![3gtgen](./img/Redstone_Redirect_3gtgen.png)

`0gt BE`, update the Piston of the upward-pushing redstone block column, the Redstone Dust **redirects and points to the block that needs to be powered**; the rail above receives the NC update provided by the column push and is activated, sending an NC update to the Piston column; the Piston responds to the NC update and extends.

`2gt TE`, the redstone block column is in position, the powered Piston completes extension, **self-checks**, finds it's **not powered**, schedules retraction.

`3gt BE`, the powered Piston starts retracting, the redstone block column starts pushing down at a slightly deeper position. **Although this powers the target Piston, the target Piston will prioritize completing the retraction action**.

`5gt TE`, reset, the target Piston self-checks again, finds it's not powered, doesn't extend again.

This way we complete a **3gt push, 3gt pull** Piston action, which is especially important in modern honey-lime tree farms.

> You should notice that if we don't give the target Piston an NC update at 0gt BE, but give it at 3gt BE, we can delay the target Piston's action by 3gt. This is the timing switching scheme currently adopted by perfect-timing all-tree-type farms: providing updates at both 0gt BE and 3gt BE. Since the target Piston is a secondary level of double recursion, pushed into position by the primary level, it won't extend due to the first powering, but instead is activated by the second powering.

### 0t Generator

This is a simple 0t generator based on redstone redirection.

![Redirect](./img/0tgen3.png)

Basically, after the redstone redirection powers the powered block and provides an NC update to make the Piston extend, **remove the powered block**, thus achieving 0t.

You can design many similar devices, including vertical stacking as mentioned in [5.4.1].

> In fact, since there's only a single Redstone Dust here, directly removing the power source of the Redstone Dust - the redstone block behind it - to create 0t won't cause much lag (approximately equivalent to 3 rails). If you're not pursuing completely dustless designs, this is also a pretty good wiring method.

The above are the basic dustless wiring methods. Let me say it again: **you going out and studying a few farms / wiring a few farms yourself is better than anything else**. Please be sure to value your own hands-on practice.
