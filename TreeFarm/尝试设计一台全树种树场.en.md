In the previous section, we completed the simplest tree farm. However, this tree farm has a serious problem: it can only process birch trees and is powerless against other tree species. To solve this problem, in this section we attempt to design a multi-species tree farm.

## Core Architecture Design

For a multi-species tree farm, we need an architecture that can meet the growth requirements of **all tree species that need to be processed**. This means **taking the union**—we need to satisfy the requirements of **the most demanding tree species** in some aspect of the architecture design. Here we temporarily only cover the five most traditional tree species: **Birch, Oak, Jungle, Spruce, Acacia**. We will explain them one by one according to the general design order.

Among them, birch has the smallest growth restrictions, so we start with spruce.

<!--这部分记得加图-->

### Spruce Architecture Requirements

Spruce requires a 5x5 area centered on the sapling with no blocks obstructing growth. For this, we can choose **triple recursion**, **pseudo-double recursion**, and **honey-slime double recursion** for trunk processing.

### Acacia Architecture Requirements

Acacia will grow **up to 2** logs on the same y-level, along the **x or z axis**, **at most 4 blocks** away from the sapling, including various cases such as straight lines and corners, as shown in the figures.

![Acacia_Straight](./img/Acacia_Straight.png)
![Acacia_L](./img/Acacia_L.png)

This means we cannot move all logs away through the main pusher, so we need to add **side branch processing**. There are two design approaches: **center consolidation processing** and **adding log outputs**.

For center consolidation processing, we need to first push the logs on the main pusher side to the center, then push the logs on the left side to the center and move them out by the main pusher, and finally push the logs on the right side to the center and move them out by the main pusher.

![HSUTF](./img/0001.gif)

A typical case of adding log outputs is nutstory's architecture. We believe you can understand how it works through the animation.

![Nutstory_Acacia](./img/type1.gif)

@Dreaming_Galaxy, XJH_Jorhai, Feng_Bl, Bright_Observer PUTF standard

> In some architectures, we will see both processing methods simultaneously, such as PUTF -2. Additionally, PUTF// provides another processing approach: adding one block to the main pusher, consolidating side branches to the middle 2 blocks and then pushing them out directly. However, this method cannot be used for cherry trees and has no significant value in GTMC's 1.20.1

### Jungle Architecture Requirements

The trunk height of jungle trees can reach up to **12 blocks**, so we need a sufficiently tall main pusher to process all logs as much as possible.

> In fact, there are multi-species tree farms that don't process full height, generally 8 blocks high (convenient for oak height limiting) and 10 blocks high (convenient for wiring), but we do not advocate this approach

The sapling drop rate for jungle trees is **1/40**, so we need to design additional leaf processing for jungle wood. The simplest method is to use **double recursion** or **honey-slime walls** to process all possible growth positions as much as possible. Generally speaking, in a top-down view, if **no more than 5 columns** of leaves within the 5x5 range cannot be processed, this tree farm can obtain sufficient jungle saplings.

![Leave_Crasher](./img/Leaves_Crasher.png)

@nutstory 5 columns cannot be processed

> Actually, this estimation method is not rigorous. Jungle leaves are distributed more densely near the trunk. Strictly speaking, we need to calculate the average processing amount for each leaf growth situation. Generally, greater than 42 is sufficient to obtain enough jungle saplings

### Oak Architecture Requirements

Since oak trees will randomly grow **large oak trees** *commonly known as drumstick trees*, we need to limit the height of oak trees to avoid this situation. The height-limiting block is located at **the 9th block above the dirt**.

> In fact, architectures and tree farms that can directly process drumstick trees do exist, but so far we cannot make them obtain sufficient jungle saplings, so they are not within the scope of discussion for multi-species tree farms

Here we use **honey-slime triple recursion** for the main pusher, **center consolidation processing** for side branches, **honey-slime walls** for leaf processing, and manually placed obsidian for height limiting.

> Students familiar with old tree farms should notice that this is almost the architecture of PTHSUTF. In fact, due to the complexity of redstone engineering design methods, the best way to learn a certain design is through "imitation". In a sense, it can be said that during the learning stage of redstone engineering, don't be afraid to "copy"—if you can copy the wiring of an excellent work once, you can learn a lot

## Block Stream to Drop Architecture Design

Due to the rather special block stream output of multi-species tree farms, we need to **reorganize** the block stream before sending it into a specially designed explosion chamber.

### Reorganization

For the design we're using as an example, all logs will be output to the front of the trunk, with at most two blocks output per operating cycle.

Students familiar with DLHSUTF/PTHSUTF/DLPTUTF should immediately think of directly using **two rows of pistons side by side** to convert the two log outputs with currently unknown intervals into a 2-wide log output. Additionally, we can use a structure similar to a **side-suction base** to convert the two log outputs with uncertain intervals into two log outputs with certain intervals (*generally 6gt*).

![BlockStream](./img/BlockStream.png)

![BlockStream2](./img/BlockStream2.png)

> Since acacia's two logs on the same y-level can be distributed at the center and within 1 block around it (as shown below), even with added log outputs, the trunk still needs to use this type of reorganization. For additional log outputs, we can simply add explosion chambers to solve their block stream to drop problem

### Explosion Chamber

Here we'll just attach some explosion chambers to process the block stream for now, but we try not to push out a 2-block-thick block stream at the same time, because **this will greatly affect the processing efficiency of the explosion chamber and the final drop recovery rate**.

> In fact, for multi-species tree farms, we can introduce two rather special explosion structures: **pure milk explosion chamber** (commonly known as "milk explosion") and **b36 explosion chamber** (commonly known as "push explosion") to handle large amounts of irregular block streams. ## Wiring—Mode Switching and Speed Limiter

Actually, after designing the timing, you just need to activate each group of pistons as strictly as possible according to your designed timing. Here we want to specifically discuss the wiring related to **mode switching**.

### Acacia Timing Switching

<!--在想这就讲时序切换会不会太复杂了-->

If you observe carefully, you will definitely notice: **Except for acacia, for all other tree species, we can push out both side honey-slime walls (*hereafter referred to as side walls*) simultaneously, thereby shortening the processing cycle**. Furthermore, we can also push out the main pusher simultaneously, further shortening the processing cycle. At this point, different operating timings emerge.

The processing solution is not complicated. You can notice that switching from normal operating timing to acacia operating timing **is essentially equivalent to adding delay to the two side walls**. Therefore, we only need to connect another line with greater delay to activate the side walls and design a device that switches between the two lines.

![Acacia_Timing](./img/Acacia_Timechanging.png)

### Automatic Oak Height Limiting

Many people say manually placing obsidian is too troublesome. Of course there is a solution: automatic height limiting.

Since different tree farms have very different automatic height-limiting methods, here we only explain the automatic height limiting used by the architecture in our example.

You can see that we only need to place a **block that cannot be pushed by the main pusher** in the corresponding position to achieve the height-limiting effect. However, in order to send this block down through some method (usually a flying machine), we can only use pushable blocks. The solution is to fill the main pusher's pushing direction with logs and insert an **activated piston** on the reorganization side to obstruct the movement of this block in the main pusher's direction.

### \* Jungle and Acacia Height Increase (1.14 and below content)

In version 1.14 and below, the growth detection for jungle and acacia is **3x3 for trunk, 5x5 for canopy**. Therefore, we can insert some blocks within the 5x5 area below a certain height (*generally 5 blocks above dirt*), thereby **forcing height increase** of jungle and acacia below a certain height to a certain height, thus improving the efficiency of these two tree species.

> Since the height increase mechanism actually prohibits jungle or acacia below a certain height from growing, it will increase bone meal consumption. Additionally, even in 1.20.1, dark oak also retains the same growth requirements (centered on the northwest corner sapling), so we can still "increase height" for dark oak

### Speed Limiter

Actually, there's still a problem here. Because trees may grow before our architecture fully resets and then trigger the detection structure, this will break our tree farm. So we need to add a speed limiter to the detection structure, and the working time of the speed limiter must also be determined according to the operating timing (reset time) of other parts of the tree farm in each mode.

For qc detection, we can directly power the retraction piston, or stick something on the structure that pushes out after detection, preventing the detection structure from pushing out before a certain moment, such as PTHSUTF's solution.
