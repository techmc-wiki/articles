If nothing unexpected happens, the tree farm you designed in the previous chapter has an efficiency of just over 10,000, with a processing cycle easily exceeding 30 or even 40gt.

What causes this? Let's analyze it step by step.

## Before We Begin — 0-tick

In [1.1.2](./前置知识与树场的基本结构.md), we mentioned that Minecraft's timing is divided into **inter-tick timing** and **intra-tick timing**. If we can implement functionality that was originally achieved through inter-tick timing **using intra-tick timing**, we successfully speed up a tree farm. Among these, the most important implementation method is the **piston action and wiring pattern** known as **0-tick**.

### The Concept of 0-tick

In [1.1.2](./前置知识与树场的基本结构.md), we also mentioned:

> BE components (pistons and note blocks) in BE execute operations in the order of **receiving updates and confirming the need to change state**. This order is sometimes called **depth**. Note blocks do not increase depth (i.e., delay the execution order of objects they update within BE).

So imagine a piston:

At `0gt BE`, a certain depth receives a **rising edge** signal, and after it extends, it receives a **falling edge** signal at a deeper depth (deeper depth means later in the BE execution queue).

Obviously, it should extend when powered and retract when depowered. This completes a **0-tick action**. In particular, for sticky pistons, the first block it pushes will arrive at a depth **"one piston deeper" than the falling edge signal**, while pulling is consistent with normal behavior.

> In fact, using the depth added by pistons as a depth unit is not rigorous. Due to differences in piston push order, when extending, it's actually possible to create depth differences smaller than "one piston" by creating b36 order. Sometimes "coral" is used as a unit (because the most widespread use seems to be in coral-based TNT duplication). This is sometimes useful, but doesn't have a major impact on overall wiring.

### Common 0-tick Generators

The idea is very simple: within **the same gt before BE ends**, give the piston a rising edge at an earlier position and a falling edge at a later position, and we can obtain a 0-tick action.

For example, this very classic 0-tick generator based on **the execution order difference between comparators and repeaters within TT**:

![0tgen1](./img/0tGen1.png)

Since repeaters generally have priority over comparators, at 2gt TT, BE will be added to the 0-tick piston first, then to the piston that removes power, so the piston that removes power must act later than the 0-tick piston, thus creating a **falling edge signal later than the 0-tick piston extension**, completing the 0-tick.

Another very classic 0-tick generator based on **Redstone Dust** (_hopefully you still remember that Redstone Dust is instantaneous_):

![0tgen2](./img/0tgen2.png)

At `0gt NU`, the redstone block is placed. At `1gt BE`, the sticky piston that removes the wire-pressing block moves. Since the sticky piston that removes the redstone block is QC'd, **BE is added only after the sticky piston that removes the wire-pressing block updates it**, thus creating a rising edge at **1 piston depth** and a falling edge at **2 piston depth** at `1gt BE`. Such a signal is sufficient for the powered piston to complete 0-tick. *Of course, you can also add depth to the falling edge through more QC pistons that update along the way, thereby activating pistons at the output end that can only be QC'd and are updated by other pistons.*

> It's worth noting that Redstone Dust turning only has PP updates. We need additional NC updates to the target piston being powered to complete the rising edge signal (that is, for pistons, rising edge and falling edge signals actually refer to receiving NC updates and confirming the need to extend or retract).
>
> ![0tgen3](./img/0tgen3.png)
>
> Actually, the 0-tick here refers to the depth at which the normal piston updates the sticky piston, not the depth of the Redstone Dust turning (although there's no big difference here).

## "Base"

For high-speed tree farms, we must largely design **bonemealing**, **detection**, **trunk processing**, and **sapling cycling** as an integrated unit, called the **base**.

### Trunk Processing

This time we must first choose the trunk processing mode.

In [1.2.2](./前置知识与树场的基本结构.md), we mentioned:

> Trunk processing generally has four methods: processing together with the main trunk, **upward push**, **downward pull**, and **side pull**.

If you think carefully, you'll find that processing together with the main trunk means **we can only use one dispenser for bonemealing**, **upward push** allows us to plant trees at the earliest at **6gt** of tree farm operation, **downward pull** is **3gt**, and **side pull** can be as fast as **0gt**. In high-speed tree farms, we generally choose **downward pull** or **side pull**.

> In fact, the problem with upward push is not just slowness. Remember that jungle trees can grow up to 12 blocks tall? Including dirt, that exceeds the piston push limit. Remember acacia side branches? Upward push causes up to three logs at the same y-level, greatly increasing processing difficulty, so upward push is now basically abandoned.

## Core Architecture — Trunk and Leaves Processing

> This is the part that determines the minimum operating cycle of a tree farm. Please carefully consider it.

Before designing the architecture, we need to first understand how to analyze the timing of an architecture.

### Timing Design

For a given architecture, we need to analyze its method of processing trees and design the shortest operating timing.

Taking the PTHSUTF architecture as an example, let's try to design the shortest processing timing for acacia (review processing principles in [3.1.2](./尝试设计一台全树种树场.md)):

![PTHSUTF](./img/PTHSUTF.png)

@Fallen_Breath, CCS_Convenant

`0gt` Main push and side wall first level extend _Perform first-level centering on all side branches_
`3gt` Main push and side wall first level in place, main push second level extends _Let main trunk direction side branches center first to avoid conflicts_
`6gt` Main push second level in place, left side wall second level extends _Center side branches still not pushed out to the center and right one block_
`9gt` Left side wall second level in place, activate main push third level and begin retracting, making main push third level 0-tick; then, right side wall second level extends _Clear side branches located in the center and center side branches in the right one block to the center_
`12gt` Right side wall second level in place, activate main push third level and begin retracting, making main push third level 0-tick _Complete processing, begin reset_
`15gt` Main push second level begins retracting, side wall first level begins retracting
`18gt` Main push first level begins retracting
`21gt` Reset

> In tree farms, we can generally consider time in units of 3gt, although this is incorrect. In fact, this architecture's acacia timing only needs to satisfy: at 6gt the first side wall second level extends, at 9gt one side wall retracts and the other side wall extends, to complete 21gt reset. This is not the only timing design method.

Performing timing design on a given architecture has strong reference value for actual wiring. At the same time, it also helps understand where the **culprit** causing a certain architecture's long operating cycle is, enabling targeted optimization.

### Architecture Design

If after performing timing design on existing architectures, you find it's still insufficient to reach the expected speed in your mind, you need to design **faster architectures**.

Here are two classic speed-up solutions:

- Nutstory architecture: In [3.1.2](./尝试设计一台全树种树场.md) we mentioned that acacia side branches can be handled not only by centering, but also by **increasing log output**. Now that we've learned timing analysis, you should notice: this architecture doesn't need to design timing for side branch centering, so it can run at 12gt, much faster than the PTHSUTF architecture.

![Nutstory](./img/Nustory_Layout.png)

@Nutstory

- TT1998's 6gt/Shixiong's 4gt architecture: They are actually the same architecture. We've already done timing analysis above. Additionally, at 4gt, **the second-level piston of the pseudo-double-recursion has already been removed and created b36 at the target position**, saplings can grow normally, so as long as we make another set of pseudo-double-recursion on the other side and solve the sapling problem, we can easily obtain a 4gt tree farm.

## Wiring

### Wiring Methods

Here we mention the simplest and most useful high-speed tree farm wiring method:

This is a very classic **double-edge 0-tick generator** based on Redstone Dust and depth (i.e., both rising and falling edges of the input signal are converted into a 0-tick signal). We can connect a sticky piston to its output, pushing and pulling a redstone block, thus obtaining a signal with **the same macro timing** as the input that **doesn't increase macro delay but has a deeper rising edge**. Using this, we can easily control wiring depth: **just connect the Redstone Dust that activates pistons in sequence according to the designed depth order in the same way**.

![Doubleside](./img/Doubleside_0tgen.png)

[Classic example based on this wiring method](https://www.bilibili.com/video/BV17i4y157vh)

### Wiring Inspection

Although we keep saying "economic base determines superstructure," we can't design a 6gt architecture and end up with wiring running at several dozen gt, right? Analyze your wiring and see **exactly where the timing problem is that's slowing down the overall tree farm operating speed**.

However, for high-speed tree farm wiring, we can't give you a more specific method. **Examining/wiring several yourself is better than anything else**.
