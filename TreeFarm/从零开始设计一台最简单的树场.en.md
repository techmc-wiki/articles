The design process for tree farms generally consists of **architecture design** and **wiring**. Here we'll use a very simple Birch tree farm as an example to guide you through designing a tree farm from scratch.

## Architecture Design

**Architecture**, in tree farms, generally refers to the **specific arrangement** of the tree farm structure. It can generally be divided into **core architecture** and **processing architecture** *i.e., the architecture design for block flow and drop collection modules*

Core architecture generally includes **bonemealing architecture**,

![bmline](./img/bmline.png)

**trunk processing architecture**,

![main_push](./img/MainPush_Layout.png)

**leaves processing architecture**,

![leave](./img/leave.png)

and **sapling recycling architecture**,

![sapling](./img/Saplingrecyc.png)

Generally, the **bonemealing structure** or **trunk processing architecture** is designed first, followed by the **leaves processing architecture** and **sapling recycling architecture**

The basic form of trunk processing architecture has been explained in the previous chapter. Here we choose **direct push** for trunk processing (_trunk processing that uses a push-out structure is called "main push"_), with the root processed together with the trunk. For bonemealing architecture, we use a **single dispenser**.

![layout_prac](./img/layout_prac.png)

Generally speaking, we don't want tree farms with processing cycles longer than 4gt to use clock activation. Therefore, we need to introduce a **detection** structure to detect tree growth and trigger the tree farm's mechanical structure

Basic detection methods include: **Comparator detection**, **QC detection**, **BUD detection**, **push limit detection**

Comparator detection:

![Comparator_Detection](./img/Comparator_Detect.png)

QC detection:

![QC_Detection](./img/QC_Detection.png)

BUD detection (here BF was lazy and directly used leaves detection):

![Leave_Detection](./img/Leave_Detection.png)

Push limit detection:

![PLD](./img/TreeFarm_PushLimitDetect.png)

Here we use QC detection. The basic principle of QC detection is that since trunks can be powered, after the sapling grows, the tree root becomes powered, and the piston diagonally below is QC'd. It just needs an update to push out, thereby triggering the tree farm's processing structure

> Although we won't introduce it yet, push limit detection is currently the most important detection method. After this chapter, almost all detection-based tree farm designs we analyze use push limit detection, which readers should note

For leaves processing architecture, Birch only needs 20 leaves, so a simple piston wall can meet the requirements. Since pistons are laggy, here we change to a **honey-slime wall** with trapdoors and chains attached *this increases the space for sapling splashing, thereby maintaining a lower leaves processing volume and largely avoiding the use of hopper minecarts*

> The lag caused by unoptimized mechanical-electrical design is quite significant, with one of the biggest culprits being pistons. Readers should always consider whether pistons in the tree farm can be reduced (another major culprit is Redstone Dust, which we'll discuss methods to reduce in another chapter)

> For more complex tree farms, the timing of architecture operation must also be designed simultaneously with the core architecture. Here we'll temporarily ignore it and leave it for detailed discussion when dealing with branch trees

For sapling recycling architecture, we cover the bottom of the tree farm with hoppers and connect them to a dropper. When the player is AFK, standing next to this dropper allows them to receive the saplings recycled by the tree farm. Additionally, we enclose the area around the core to avoid excessive sapling splashing range

For processing architecture, we use a simple **TNT explosion chamber**

> It's worth noting that since current high-speed all-tree-type farms have multiple log block flow outputs, the processing architecture will also be very complex. We'll discuss general design approaches for these situations in the analysis of PUTF std2.

## Wiring

The basic idea is to connect the various parts of the tree farm without affecting the tree farm's timing

First, we connect a **4gt clock** to the bonemealing dispenser and sapling dropper

![BM+Clock](./img/BM+4gtclock.png)

Then use Redstone Dust to activate the pistons in each part

For the TNT explosion chamber, we simply place a **TNT duplicator**. To reduce space usage, we use water flow to **buffer TNT** and use water flow to collect drops

![TNTDUPE](./img/TNT复制与水流缓存.png)

At this point, a most basic tree farm design is complete. If you didn't reference the illustrations, this can be said to be the first tree farm you've designed. Now try analyzing its timing?

> Timing analysis is the most basic method of analyzing tree farms. Reading the tree farm's operation timing through architecture and wiring is an essential basic skill for every tree farm enthusiast.
