# Block Entities

## Concept and Content of Block Entities

Blocks themselves store data through predefined, limited BlockStates; render models through default block rendering behavior; and execute logic through update systems, scheduling systems, etc.
However, this pattern has limitations and cannot directly satisfy the functionality of certain blocks in game interactions.

Block entities were created for this purpose. They provide three important additional features for ordinary blocks:
- **_Store data using NBT_**
- **_Custom rendering behavior_** 
- **_Update during every tick_**

### Content of Block Entities

Block entities vary by block type, but an abstract, basic block entity contains the following basic information and functionality:

- type
- world (the world it exists in)
- pos (position)
- removed (whether it has been removed)
- cachedState (cached state of the corresponding block)

### Block Entities and Blocks

Block entities are bound to blocks in the world. In a world, each block position can only have one block entity instance.

Only blocks that have been specially declared can legally coexist with specified types of block entities.
Block entities are added and removed together with blocks. However, through special means such as **update suppression**, it is possible to retain a block entity when its block is removed.


## Important Blocks with Block Entities

### Hopper

In Minecraft, the hopper is an important block entity primarily used for automated item transfer.

#### Information Carried by Hopper Block Entity

A lovely hopper block entity carries the following basic information:

- `inventory` storage space (5 slots)
- `transferCooldown` cooldown (default is -1)
- `lastTickTime` world time of the last tick
- `facing` direction

#### Functionality of Hopper Block Entity

- **Transfer items to target container**
- **Extract items from container**
- **Attempt to pick up item entities**

#### Working Process of Hopper Block Entity

Every game tick, the hopper executes the following steps:

1. **Decrease cooldown time (cd, Cooldown)**
   - If cd is still greater than 0, the hopper will not perform any pull or output operations, but will directly end the calculation.
   - If cd reaches zero, the hopper will begin attempting to execute **output** and **pull** operations.

2. **Execute item transfer logic**
   - **First attempt output** (place items into target container)
   - **Then attempt pull** (extract items from container above)
   - **If extraction or output succeeds**, the hopper will reset the cooldown time and synchronize its own data.

![img.png](img/HopperFlowchart-basic1.png)

Now, let's move on to the ***specific logic of attempting to pull and output***.

When the hopper determines it needs to do something, it will first attempt to output to the container it's pointing at. Then, it will attempt to extract items from above. Finally, if either extraction or output succeeds, it will set the cooldown and update and synchronize its own data.

![img.png](img/HopperFlowchart-basic2.png)

Let's look at the output and pull logic separately.

First, let's look at ***output***...
The ***output*** operation is completed by the `insert()` method, which transfers items from the hopper to the target container and returns a boolean value to indicate whether the transfer was successful. If the transfer succeeds, the hopper enters a cooldown state.

**Basic logic of output:**

1. **Confirm whether the target container is available**
   - If the target container is full, output fails.
   - If the target container has available slots, attempt to transfer items.

2. **Attempt to output items**
   - Iterate through the item slots in the hopper, attempting to transfer items to the target container one by one.
   - As long as one transfer succeeds, return "success"; otherwise return "failure".

![img.png](img/HopperFlowchart-basic3.png)

When calling the `transfer()` method to attempt item transfer, the `transfer()` method will return an item stack:

- If the return value is empty, it means the items have been completely transferred, and output succeeds.
- If the return value is not empty, it means the items could not be fully transferred, and output fails.

![img.png](img/HopperFlowchart-basic4.png)

At this point, the output part is complete. Next, let's look at the ***pull*** part.
The logic for pulling items is similar to output, but in the opposite direction:

- During **output**: the hopper `transfer()` items to the **target container**.
- During **pull**: some **container** `transfer()` items to the **hopper**.

Since the logic of the `transfer()` method will be explained in detail shortly, it will not be repeated here.

![img.png](img/HopperFlowchart-basic6.png)

`transfer()` is the core method for implementing item transfer in hoppers. It receives the following parameters:

- **from**: item source (can be a hopper or other container)
- **to**: transfer target (can be a container or another hopper)
- **itemStack**: the item stack to be transferred
- **slot**: target slot in the target container

Execution flow is as follows:

1. **Check the status of the target slot**
   - If the slot is empty, directly place `itemStack` into it and clear `itemStack`.
   - If the slot already has items of the same type and still has remaining space, merge the items and clear `itemStack`.

2. **Determine whether the transfer succeeded**
   - After transfer, if `itemStack` is empty, it indicates successful transfer; otherwise it indicates failure.

3. **Special handling of timing for hopper-to-hopper transfers**
   - If the target is another hopper, the target hopper will be given `8gt` of cooldown time.
   - If the target hopper `to` has already calculated before the source hopper `from`, the target hopper `to` will only receive `7gt` of cooldown time.
     > It's worth noting that this only occurs when the target hopper is empty;
     > Additionally, even if `from` calculates after `to` and gives `8gt` of cooldown time, since `to` will immediately -1 the cooldown when it ticks subsequently, the result is still only `7gt` cooldown.

![img.png](img/HopperFlowchart-basic5.png)

At this point, you have fully understood the basic operating mechanism of hoppers. I believe this will provide great help for understanding subsequent chapters.

#### Moving Piston

Another extremely important block is called moving_piston, the moving piston, commonly known as B36.

#### Information Carried by Moving Piston Block Entity

A moving piston block entity carries the following basic information:

- `pushedBlock` stored block
- `facing` direction
- `extending` whether it is extending (related to the direction of animation playback)
- `source` whether it is the piston body during retraction
- `progress` current movement progress
- `savedWorldTime` world time when logic was last executed

#### Functionality of Moving Piston Block Entity

- **Handle piston pushing entities**
- **Update animation progress**
- **If `progress>=1`, replace MOVING_PISTON with the block stored in `pushedBlock`**
- **Render animation on the client**

## [!ADVANCED] Timing Between Block Entities

The block entity `tick` process execution is maintained by the following two lists in `LevelChunk`:

- `pendingBlockEntityTickers`: block entities dynamically added during the `tickBlockEntities` process.
- `blockEntityTickers`: block entities to be `tick`ed

Both of these lists are ArrayLists, so their traversal order completely depends on the order in which elements are added. The specific process is as follows:

- `tickBlockEntities()` will first add elements from `pendingBlockEntityTickers` to the end of `blockEntityTickers`, then clear `pendingBlockEntityTickers`.
- When traversing `blockEntityTickers`, if a `TickingBlockEntity` has been removed (`isRemoved()` returns `true`), it is removed from the list; otherwise its `tick()` method is called for updating.

When adding a new block entity, the `addBlockEntityTicker()` method is called to add it to the list. If currently in the `tick` process (`tickingBlockEntities == true`), it is added to `pendingBlockEntityTickers`; otherwise it is directly added to `blockEntityTickers`.

During unloading, block entities within the same chunk are saved in `Map<BlockPos, BlockEntity>`. During loading, this Map is traversed and `addBlockEntityTicker()` is called sequentially to add them to `blockEntityTickers`. Therefore, their execution order depends on the calculation order of `BlockPos`'s `hashCode`.

The loading order of block entities between different chunks depends on the chunk loading order, i.e., whichever chunk loads first will have its block entities added to `blockEntityTickers` first.

**Summary in one sentence**:

Before unloading, the calculation order of block entities depends on the placement order. After reloading, the calculation of block entities within a chunk depends on the hash of their pos, while block entities in different chunks depend on the chunk loading order.

Related code is as follows (some irrelevant content has been removed):

```java
// Used to temporarily store block entities dynamically added during the tickBlockEntities process
private final List<TickingBlockEntity> pendingBlockEntityTickers = Lists.newArrayList();

// List of block entities to be ticked
protected final List<TickingBlockEntity> blockEntityTickers = Lists.newArrayList();

private void tickBlockEntities() {

   this.tickingBlockEntities = true;

   if (!this.pendingBlockEntityTickers.isEmpty()) {
      this.blockEntityTickers.addAll(this.pendingBlockEntityTickers);
      this.pendingBlockEntityTickers.clear();
   }

   // Traverse the list of block entities being ticked
   Iterator<TickingBlockEntity> iterator = this.blockEntityTickers.iterator();
   while (iterator.hasNext()) {
      TickingBlockEntity tickingBlockEntity = iterator.next();

      if (tickingBlockEntity.isRemoved()) {
         // If the block entity has been removed, remove it from the list
         iterator.remove();
      } else if (this.shouldTickBlocksAt(tickingBlockEntity.getPos())) {
         // If the position should be ticked, call the tick() method for updating
         tickingBlockEntity.tick();
      }
   }

   this.tickingBlockEntities = false;

}

// Add a block entity's ticker
public void addBlockEntityTicker(TickingBlockEntity tickingBlockEntity) {
   (this.tickingBlockEntities ? this.pendingBlockEntityTickers : this.blockEntityTickers)
           .add(tickingBlockEntity);
}

// All block entities within the chunk
protected final Map<BlockPos, BlockEntity> blockEntities = Maps.newHashMap();

// After chunk loading is complete, register all existing block entities (add them to the tick system)
public void registerAllBlockEntitiesAfterLevelLoad() {
   this.blockEntities.values().forEach(blockEntity -> {
      // Add a ticker for this block entity, i.e., add it to blockEntityTickers
      this.updateBlockEntityTicker(blockEntity);
   });
}

```
