### Phase Division of Micro-Timing

**1gt** is like **one day** in the real world, and within a day, we have a certain order of arrangements: waking up, breakfast, lunch, dinner, sleeping... In Minecraft, timing with 1gt as the smallest unit is called macro timing, while the more subdivided timing within a certain 1gt is called micro-timing.

In the previous text, we know that MC always executes things in a priority order. Through source code, experiments and other means, we discovered—Minecraft always roughly executes fixed events in the following order within 1gt:

1. **<WTUColor>World Tick Update</WTUColor>**, abbreviated as <WTUColor>WTU</WTUColor>. There is a <WTUColor>"timer"</WTUColor> bound to the world in the game. When the world is created, the <WTUColor>"timer"</WTUColor> is initialized to 0. During <WTUColor>**this game phase**</WTUColor>, the <WTUColor>"timer"</WTUColor> increments by 1. We call the nth tick: the tick of the <WTUColor>**World Tick Update phase**</WTUColor> that makes this <WTUColor>"timer"</WTUColor> increase to n.
2. **<TTColor>Schedule Tick / Tile Tick / Next Tick Entry\*</TTColor>**, abbreviated as <TTColor>TT</TTColor>/<TTColor>NTE*</TTColor>. Most components with delayed execution behavior are controlled by <TTColor>"Scheduled Tick"</TTColor> and executed during the <TTColor>**Scheduled Tick phase**</TTColor>. *(The mention of NTE and the **incorrect** translation of Next Tick Entry here is intended to help readers understand some early documents, and we do not wish for you to continue using these two names.)*
3. **<CTColor>Chunk Tick</CTColor>**, abbreviated as <CTColor>CT</CTColor>. Chunk Tick is divided into **lightning, snow** and **Random Tick**. During this phase, the game first processes **lightning, snow** and other events; then processes **Random Tick**: when a player is within a certain distance, <CTColor>"crop growth"</CTColor>, <CTColor>"grass block spread"</CTColor>, <CTColor>"water freezing"</CTColor> and other random tick events occur. In each <CTColor>**Chunk Tick phase**</CTColor>, the game traverses all chunks near the player, then randomly selects blocks in these chunks to execute these events.
4. **<BEColor>Block Event</BEColor>**, abbreviated as <BEColor>BE</BEColor>. Things like **piston pushing/pulling, note blocks making sounds** happen in this phase. Taking pistons as an example, when a piston finds that its [**actual state**] does not match its [**powered state**], it will add a <BEColor>"**Block Event**"</BEColor>. <BEColor>"**Block Events**"</BEColor> added during the <BEColor>**Block Event phase**</BEColor> will be executed in this <BEColor>**Block Event phase**</BEColor>, while <BEColor>"**Block Events**"</BEColor> added in phases outside the <BEColor>**Block Event phase**</BEColor> will be placed in the nearest future <BEColor>**Block Event phase**</BEColor> for execution. Continuing the piston example, the piston will place moving blocks (b36) at the intended positions of blocks to be pushed/pulled during <BEColor>**this phase**</BEColor>.
5. **<EUColor>Entity Update</EUColor>**, abbreviated as <EUColor>EU</EUColor>. Entities actively **perform movement, entity AI and other behaviors** every tick. All **entity behaviors**, such as <EUColor>"creature movement"</EUColor>, <EUColor>"TNT explosion"</EUColor>, <EUColor>"monster attacks"</EUColor>, occur in <EUColor>**this phase**</EUColor>. Non-player <EUColor>"activation"</EUColor> of pressure plates and tripwires, causing them to produce redstone signals, also occurs in <EUColor>this phase</EUColor>.
6. **<TEColor>Block Entity / Tile Entity</TEColor>**, abbreviated as <TEColor>TE</TEColor>. Some blocks need to **run their own related logic** every tick, and these things happen in the <TEColor>**Block Entity phase**</TEColor>. During the <TEColor>**Block Entity phase**</TEColor>, hoppers will <TEColor>"absorb"</TEColor> and <TEColor>"transfer"</TEColor> items; blocks pushed by pistons become moving blocks (b36), which will perform <TEColor>"push entity"</TEColor> logic during the first two <TEColor>**Block Entity phases**</TEColor> after their creation, and become normal blocks in the following third <TEColor>**Block Entity phase**</TEColor>.
7. **<ATColor>Async Task / Network Update</ATColor>**, abbreviated as <ATColor>AT</ATColor>/<ATColor>NU</ATColor>, also called Player Action. <ATColor>"Player actions"</ATColor> are actually network packets sent from the client to the server. During <ATColor>**this phase**</ATColor>, at the end of each tick, the server will uniformly execute all "player action" packets received during this tick.

These are the main game phases involved in regular analysis, and we will also expand on these events in the following text.

### Instant

In the above text, we recognized that 1gt is further divided into different phases. Just like breakfast can only be eaten in the morning and lunch can only be eaten at noon, many events in Minecraft can only follow a certain order and occur in specific phases. But there are also some "immediate response" events, just like when we're thirsty we can immediately drink water, regardless of whether it's morning, noon, or evening.

If a block's behavior can occur in any phase and is only triggered by **block updates**, then this block is called an **instant component**.

For example, redstone dust turning on/off is instant. If a player turns off a lever, it can cause redstone dust to turn off in the player action phase; if a piston pushes away a redstone block, it can cause redstone dust to turn off in the block event phase.

### Delayed

Since there are instant events, there must be "delayed" events. For example, if a player places a redstone block and the redstone block activates a repeater, this repeater will light up after 2gt. The repeater lighting up has a **delay** and is **controlled by the game** (controlled by scheduled ticks), occurring in a **specific phase** (Scheduled Tick phase), so the repeater is not an instant event.

In the following sections, we will expand in detail on these **different game phases**.

### Operating Phases of Common Components

|                Component Type                |           Operating Phase            |
|:--------------------------------------------:|:------------------------------------:|
|            Command block runs command            |  <TTColor>Scheduled Tick TT</TTColor>  |
| Repeater, Comparator, Redstone Torch, Observer on/off |  <TTColor>Scheduled Tick TT</TTColor>  |
|          Redstone Dust, Rails change state          |             Instant              |
|         Fence Gates, Trapdoors change state         |             Instant              |
|         Hopper changes state due to redstone signal         |             Instant              |
|           Hopper absorbs, transfers items           | <TEColor>Block Entity TE</TEColor> |
|      Note Block, Bell changes state due to redstone signal      |             Instant              |
|           Note Block, Bell makes sound           | <BEColor>Block Event BE</BEColor> |
|    Dispenser, Dropper changes state due to redstone signal    |             Instant              |
|       Dispenser dispenses/Dropper drops items        |  <TTColor>Scheduled Tick TT</TTColor>  |
|              Redstone Lamp lights up              |             Instant              |
|              Redstone Lamp turns off              |  <TTColor>Scheduled Tick TT</TTColor>  |
|        Button, Pressure Plate, Tripwire lights up        |           Instant[^1]            |
|        Button, Pressure Plate, Tripwire turns off        |  <TTColor>Scheduled Tick TT</TTColor>  |
|       Falling block determines fall, creates entity       |  <TTColor>Scheduled Tick TT</TTColor>  |
|           Falling block falls, lands           | <EUColor>Entity Update EU</EUColor> |
|             Piston extends or retracts             | <BEColor>Block Event BE</BEColor> |
|            b36[^2] pushes entity            | <TEColor>Block Entity TE</TEColor> |
|              b36 naturally lands              | <TEColor>Block Entity TE</TEColor> |
|       b36 retracted and landed by sticky piston[^3]       | <BEColor>Block Event BE</BEColor> |

[^1]: However, pressure plate lighting can only be triggered by entity movement, so it can actually only occur in Entity Update or Player Action; similarly, buttons can only be pressed by players or arrows, so they are also limited to these two phases.

[^2]: b36: minecraft:moving_piston, moving piston.

[^3]: When a sticky piston retracts and there is b36 in front, it will immediately land it, which is the so-called sticky piston short pulse.

### Basic Analysis of Intra-Tick Timing

Here, we give a basic example.

In the figure below, all command blocks contain the command time query gametime

![Basic Analysis](./img/MicroTickIntro_Basic_Analysis.png)

Rising edge _(Pull down lever, sticky piston extends)_:

| Tick | Phase                  | Event                             |
|------|------------------------|-----------------------------------|
| 0    | <ATColor>AT</ATColor> | Player pulls down lever                     |
| 0    | <ATColor>AT</ATColor> | First piston adds block event           |
| 1    | <BEColor>BE</BEColor> | First piston extends                   |
| 3    | <TEColor>TE</TEColor> | First redstone block lands                 |
| 3    | <TEColor>TE</TEColor> | Command block schedules to run at tick 4 (1 tick delay) |
| 3    | <TEColor>TE</TEColor> | Second piston adds block event           |
| 4    | <TTColor>TT</TTColor> | First command block reads 4              |
| 4    | <BEColor>BE</BEColor> | Second piston extends                   |
| 6    | <TEColor>TE</TEColor> | Second redstone block lands                 |
| 7    | <TTColor>TT</TTColor> | Second command block reads 7              |

Falling edge _(Pull back lever, sticky piston retracts)_:

| Tick | Phase                  | Event                   |
|------|------------------------|-------------------------|
| 0    | <ATColor>AT</ATColor> | Player turns off lever           |
| 0    | <ATColor>AT</ATColor> | First piston adds block event |
| 1    | <BEColor>BE</BEColor> | First piston retracts         |
| 1    | <BEColor>BE</BEColor> | First redstone block is removed       |
| 1    | <BEColor>BE</BEColor> | Second piston adds block event |
| 1    | <BEColor>BE</BEColor> | Second piston retracts         |
| 3    | <TEColor>TE</TEColor> | Both redstone blocks land         |
| 4    | <TTColor>TT</TTColor> | Both command blocks read 4    |
