# Block Events


This article discusses the implementation principles of Block Events (BE).

**Basic Section**

- The concept and purpose of Block Events
- Information structure of Block Events and related components
- A practical understanding of Block Event Delay (BED)

**Advanced Section**

- Block Event queue and breadth-first search mechanism

## What are Block Events?

In Minecraft gameplay interactions, to reduce network synchronization pressure between the server and client, the game avoids synchronizing massive amounts of data from all complex blocks to the client every tick.

Instead, the server chooses to tell the client: "A specific event happened here," and then lets the client's blocks simulate what happened on the server locally.

**Block Events** were created for this efficient "notification and simulation" approach.

### Information Structure of Block Events

When a Block Event is created, it contains and only contains the following four core pieces of information:

- Position `pos`: where the event occurs
- Block type `block`: which type of block is executing this event
- Type `type`: which specific event action (for pistons, represents extend or retract)
- Data `data`: necessary parameters attached when executing the event (for pistons, represents the direction of push)

## Block Event Delay (BED)

When multiple blocks are activated at the same time and generate Block Events, do they execute simultaneously?

Minecraft's worldview tells us: there is no absolute "simultaneity" in the game. Even within the same game tick, event execution must have a sequential order. This phenomenon caused by the time difference in execution order is called **Block Event Delay (BED)**.

Here's how Block Event Delay manifests in the game:

![Observation of Block Event Delay](./img/BlockEventDelay_StandardExemplification.png)

When we flip the lever, we find that when changing the position of the redstone block, the number of pistons extending nearby changes. When the redstone block is closer to the pressure line block, the number decreases, and when the redstone block is farther from the pressure line block, the number increases.

### Practical Understanding of Depth and Queue

You can imagine the execution of Block Events as people queuing to buy tickets: **first come, first served**.

When a Block Event executes, it may trigger subsequent new events. These new events are appended to the end of the queue. To track which "generation" these events were produced by, we introduce the concept of **depth**: initial events have depth 0, events directly produced by them have depth 1, and so on, used to characterize the derivative causal chain between events.

Below is an example of a 0t bottom retraction base in a tree farm, demonstrating how Block Event depth is applied in actual design:

![Galaxy Brand Bottom Retraction](./img/BlockEventDelay_Bottom_Retraction.png)

0gt AT: Lever is pulled down

1gt BE depth 0: Sticky piston starts retracting

1gt BE depth 1: Sticky piston pulls back podzol, redstone dust redirects

1gt BE depth 2: Bottom retraction piston is updated, self-checks, finds itself activated, extends

1gt BE depth 3: Powered block is removed, updates bottom retraction piston, self-checks, finds it should retract, adds retract Block Event, finds itself extending, 0t occurs.

![Galaxy Brand Bottom Retraction_gt4](./img/BlockEventDelay_Bottom_Retraction_gt4.png)

3gt TE: All blocks are in place.

4gt BE depth 0: Top and bottom regular pistons extend, dirt's sticky piston adds Block Event

4gt BE depth 1: Regular piston extends, updates sticky piston, pushback sticky piston extends, base activates piston self-check. Redstone block is removed.

4gt BE depth 2: Top and bottom regular pistons retract, send updates, update dirt's sticky piston, sticky piston self-checks, finds it shouldn't extend, adds retract Block Event, bottom sticky piston same logic

4gt BE depth 3: Redstone block sticky piston and podzol sticky piston's 0t occurs, podzol instantly in place. Redstone block in place, sticky piston controlling bottom retraction piston's powered block extends.

6gt TE: Redstone Dust is redirected, powered block in place, base completes reset.

## [!ADVANCED] Block Event Queue and Breadth-First Search

At the source code level, the "queuing" of Block Events is essentially a breadth-first search queue mechanism.

In the game's main loop, Block Events are stored in a first-in-first-out queue. When a source point (such as a central piston activated by a lever) sends updates to the surroundings in NC update order, the surrounding pistons will sequentially push their Block Events to the end of the queue.

During execution, the game takes an event from the head of the queue to execute:

1. Take out and execute the first node with depth `0`
2. If this update triggers new updates, the newly generated Block Events (depth `1`) are pushed to the very end of the current queue in update order.
3. The game continues executing the next node with depth `0` in the queue, until all depth `0` events are dequeued, then it starts executing depth `1` events.

This operation is completely equivalent to breadth-first search.

Here's an example:

![BED_BFS](./img/BlockEventDelay_BFS.png)

Where the long edge extends from +x to -x direction

When the lever is pulled down, the central piston is activated and updates the three surrounding pistons in the order of -x, +x, +z, pushing them into the queue sequentially. The game then executes the pistons in -x and +x directions in order, then updates the next piston in the +x direction. At this point, a Block Event is added and pushed into the queue, located after the piston in the +z direction. Then the piston in the +z direction is executed, and finally the last piston in the +x direction is executed.

We can abstract each piston as a node and treat pistons with depth 0 as root nodes. Note that in actual situations, this graph may have multiple source points. We perform multi-source BFS simulation on this abstracted "piston graph," where the order in which child nodes join the queue is the NC update order, determined by the direction of child nodes relative to parent nodes. Child nodes that have already been visited will not be visited again.
