# Tick and Inter-Tick Timing


> We have already learned about update theory in the previous article, but curious Little B decided not to stop there. He took out the ~~Redstone Repeater~~ Repeater from the creative inventory and pulled down a lever.
>
> Perhaps Little B was too sensitive, but when he pulled down the lever, he noticed that the Repeater didn't seem to light up immediately. He thought the game had a bug and tried several more times. He quickly realized this was a stable, reproducible phenomenon. Obviously, unlike the numerous instantaneous phenomena we discussed in the previous article, an **asynchronous** phenomenon appeared here, which he called **delay**.

## Game Tick, Redstone Tick, and Delay

In Minecraft, there are a large number of events that need to be processed. These events are placed in sequence in a large loop, which we call the game main loop. When not using the _/tick_ command and the game is running normally, this main loop always cycles at an interval of approximately **20 times per second**. We call this minimum interval a **GameTick**, abbreviated as **gt**. Due to historical reasons, in some cases we still use the notation **RedstoneTick**, abbreviated as **rt**. It has the following conversion relationship with gt: $1rt=2gt$.

This means:

1. 1gt = $\frac{1}{20}$ seconds = $0.05$ seconds
2. Even if Minecraft has sufficient computing power and completes all the calculations needed for this gt in less than 0.05 seconds, it will stop calculating until the next gt arrives.

Therefore, players have defined two metrics to measure whether the game is lagging: **TPS** and **mspt**. TPS stands for Tick Per Second, which refers to how many game ticks the game executes per second. Normally, this value is 20, but when the game lags, it will be lower than 20. And mspt, which stands for Millisecond Per Tick, refers to the number of milliseconds the game currently needs to spend executing each tick. The lower the value, the less the game lags.

It is worth noting that mspt always maintains the following relationship with TPS:

- When mspt is below 50, TPS is 20
- When mspt is greater than 50, TPS = $\frac{1000}{mspt}$

In the game, the vast majority of redstone components have **delayed** responses. We define the unit of a component's **macroscopic** delay as **gt**. The delays of some components are as follows:

- Repeater: 2-8gt
- Comparator: 2gt
- Observer: 2gt
- ...

## Repeater and Comparator Basics

Repeaters and Comparators have macroscopic delays. The delay of a Repeater is related to its set gear position, while the delay of a Comparator is always 2gt.

The basic function of a Repeater is to amplify signals. That is, it converts any redstone signal with a strength not equal to 0 into a signal with strength 15 after delaying the specified number of game ticks.

When a Repeater's side is directly powered face-to-face by a Repeater or Comparator, it enters a "locked" state. At this time, any changes to its input end do not affect its lit or extinguished state until the side input ends. After waiting for the delay corresponding to gt, if the state needs to be changed, it changes its state.

A Comparator has a total of 3 input ends: the front and two side inputs. When there is no input on the sides, it always directly outputs a signal with the same signal strength as the front input. When there is input on its sides, the Comparator's behavior depends on its mode. The Comparator has two modes:

- Compare mode
- Subtract mode

---

Its default is **Compare mode**. When the Comparator is in **Compare mode**, it always follows the following logic for output:

> Take the maximum value of the left input and right input. If its maximum value is greater than its front input, then no output is produced; otherwise, output its front input.

---

When the Comparator is in **Subtract mode**, it always follows the following logic for output:

> Take the maximum value of the left input and right input. If its maximum value is greater than its front input, then no output is produced; otherwise, output its (front input - maximum value of side inputs).

---

The interaction between Comparators and containers is not discussed here.

## Charging Theory

We establish charging theory to explain a phenomenon that occurs with components such as Repeaters and Comparators, whose important characteristic is charging Redstone Dust or other Repeaters/Comparators through blocks.

We define that for any **chargeable** block, it has two charging states: **strong charging** and **weak charging**.

When a chargeable block is strongly charged, its signature characteristic is that it can activate Redstone Dust in all 6 directions around it, regardless of its orientation. When weakly charged, it can only activate Repeaters, Comparators, and some other redstone components whose input ends face the chargeable block. This is also used as an important method to distinguish between the two types of charging.

The following methods can make a block strongly charged:

- Repeaters, Comparators, Redstone Torches, and Observers directly emit signals toward the chargeable block.

The following methods can make a block weakly charged:

- A chargeable block is pointed to by Redstone Dust with a signal strength not equal to 0.

![Strong Charging and Weak Charging](./img/Tick_StrongOrWeak.png)

In MC, chargeable blocks are the majority, so here we distinguish between the two types of blocks by the distinctive characteristic of non-chargeable blocks—not cutting lines. Cutting lines refers to a phenomenon where the vast majority of blocks will cut off vertically connected Redstone Dust. We call this phenomenon cutting lines. However, some blocks, such as glass, slabs, half-slabs, chests, etc., do not have this phenomenon, which also means they are non-chargeable blocks. We define here that Pistons/Sticky Pistons are also non-chargeable blocks.

## Inter-Tick Timing

> This section focuses on explaining basic timing analysis.

![Basic Timing](./img/Tick_Basic_Analysis.png)

Here we give an example to use the concept of gt to analyze the simplest timing.

As shown in the figure, when the lever is pulled down, the Repeater on the right will be activated after a delay of 8gt, then strongly charges the block. At this time, the redstone signal strength on the side of the Comparator in Subtract mode is 14, while the signal strength of the front input is 15. 15-14=1>0, activating the redstone lamp after a delay of 2gt, for a total delay of 10gt.

The Comparator on the left is in Compare mode. The signal strengths of its front and sides are both 14. The strengths on both sides are equal. After a delay of 2gt, it outputs a redstone signal with signal strength 14, strongly charging the block in front of it. Then the Repeater activates the redstone lamp after a delay of 6gt, for a total delay of 8gt.
