## Broad Updates

In Minecraft, blocks establish connections with each other through mutual "notifications".

One day, you had a sudden idea—you placed a note block on the ground. Then, you placed a redstone block adjacent to the note block, and the note block emitted a pleasant sound. So, how did the note block know it should emit a sound at exactly this moment? Does the note block keep sending messages to the player: Should I make a sound? Should I make a sound? Should I make a sound... This is obviously unreasonable. In fact, the note block was able to emit a sound because when the redstone block was placed, it "notified" the note block: Hey, there's a change here, pay attention.

This **notification** behavior is called a **broad update**.

It's important to note that the redstone block does not notify "I am a redstone block" or "this changed from air to a redstone block", but simply notifies: there's a change here. In other words, in Minecraft, update behavior does not contain specific information about the action.

## Types of Updates

In Minecraft, there is more than one type of update.

Let's continue with the example above. One day, you had another flash of inspiration—you placed a redstone block on the ground first, then placed a fence gate one block away, and placed a note block between the redstone block and the fence gate. You suddenly noticed that the note block didn't emit a sound. So, you right-clicked the fence gate to open it, but the note block still didn't emit a sound. Logically, both "placing a redstone block" and "opening a fence gate" are changes, and they should both "notify" the note block. Why didn't the note block emit a sound as expected? With confusion, you directly broke the fence gate, and at this moment the note block emitted a sound.

<!-- 这里应该有个动图 -->

——Why is this?

You then speculated: perhaps there's some difference between the "notifications" from "placing a redstone block" and "opening a fence gate".

In fact, these two notifications are indeed different. Nowadays we call them **NC Update** and **PP Update** respectively.

At this point you also thought: why doesn't the note block immediately emit a sound when it's placed? Doesn't it "check whether it should say something" when it's "born"? ——That's right, the note block is rather lazy, it indeed doesn't "check whether it should emit a sound".

This behavior of checking itself when being placed and in other situations is called **self-inspection**. **Self-inspection** can be viewed as a **special type of update**. And note blocks do not perform self-inspection.

In your previous studies, you also learned that: a comparator is a redstone component that can detect the number of items in a container, outputting redstone signals of different strengths from 0-15 based on the quantity of items in the container. You thought: when the number of items in a container changes, the container doesn't necessarily need to emit an **NC Update** or **PP Update**, after all, only the internal item quantity changed, which won't have much impact on surrounding blocks. But you know that comparators can still promptly change their output redstone signal strength, indicating that comparators can still receive notifications of this change. This is because when the number of items in a container changes, the container also emits a special type of notification.

Such notifications are called **Comparator Updates**.

## NC Update

### Concept and Behavior of NC Update

Blocks emit **NC Updates** when they are placed, broken, or undergo changes that significantly affect surrounding blocks. For example:

- Placement and breaking of blocks
- Changes in redstone dust power level
- Piston push/pull\*
- ...

> In the official deobfuscation, NC Update is `neighborChanged`, same as the mcp deobfuscation. In yarn deobfuscation it's called `updateNeighbors`

However, some changes do not emit **NC Updates**, the most common being:

- Connection state changes of connectable blocks, such as glass panes connecting to other blocks, various walls connecting to other blocks
- Connection state changes of redstone dust, such as redstone dust connected to north-south redstone dust also connecting to east redstone dust
- Opening/closing of trapdoors and fence gates
- Activation state changes of dispensers and droppers (**! Important**)
- Activation state changes of hoppers
- ...

> The specific behavior of pistons is not within the scope of this article.

A block that emits an NC Update is called an **update source**[^1]. For most blocks, the update source is the block itself when emitting updates.

[^1]: Understanding of **update source** is not explained extensively in the basics section, and not all NC Updates have an update source. See [Advanced Section](#172-nc更新的调用与响应).

When an update source emits an NC Update, it follows the order **West East Down Up North South**, sequentially emitting NC Updates to blocks immediately adjacent to the west, east... of the update source. Such updates are updates in the usual sense.

Some blocks have more than one update source when emitting NC Updates, with special update sources and ranges:

- Redstone dust power level changes are second-order neighbor updates[^2].
- Directional redstone components, such as redstone repeaters, comparators, and observers, when their activation state changes, first separately update the block their output end points to, then emit NC Updates with the block pointed to by the output end as the update source, and do not update themselves during NC Updates.
- When flat powered rails switch activation state, they first generate NC Updates with themselves as the update source, then generate NC Updates with the block below as the update source.
- When sloped powered rails switch activation state, the update order is up, middle, down, middle, down, up (NC Update with itself as update source abbreviated as **middle**, with block above as update source abbreviated as **up**, with block below as update source abbreviated as **down**, detailed interpretation see [Update Behavior of Sloped Rails](./特殊的更新行为.md#3.2-[!ADVANCED] 斜侧铁轨的更新行为)).

These are common and typical examples. More special cases can be found at [Wiki-Block Update](https://zh.minecraft.wiki/w/%E6%96%B9%E5%9D%97%E6%9B%B4%E6%96%B0#NC%E6%9B%B4%E6%96%B0)

[^2]: Redstone dust has more complex update behavior, not expanded in this article. ~~May add redstone dust special DLC later~~

### Properties of NC Update, QC Activation, BUD Devices

The most typical property of NC Updates is their ability to trigger **BUD devices**.

When a block's current state differs from its **intended state**, this block can be called a **BUD device**, i.e., **Block Update Detector**. This block state can also be called a **BUD state**.

The most typical BUD device is a piston that is powered (receiving a redstone signal) but not activated (not extended). Pistons have a special powering range: when the piston itself and the block above the piston (can be air) are powered, the piston can be considered powered. Powering a piston by powering the air above it is called **QC powering**. When such a powered piston extends after receiving an NC Update, it's called **QC activation**.

**QC (quasi connectivity)** is called **quasi-connectivity**. Pistons, sticky pistons, droppers, and dispensers all have this property.

Imagine a redstone block placed diagonally above a piston. At this moment, the piston is QC powered. However, the placement of the redstone block only updates the six adjacent blocks and cannot update the piston, so no block notifies the piston "there's a redstone signal here". At this point, the piston should extend, but hasn't extended because it hasn't received an NC Update. That is, *the current state differs from its intended state*, meaning this piston is in a **BUD state**, and the piston constitutes a **BUD device**.

At this point, if you place a block next to the piston, the block placement will emit an NC Update. The piston receives the NC Update and exits the BUD state.

A simplest commonly used self-resetting BUD device is as follows:

![Example](./img/01-budexample1.gif)

In the image, the redstone block **QC powers** the sticky piston below. When the sticky piston receives an NC Update and extends, the redstone block no longer powers it, and then the sticky piston retracts, and the redstone block QC powers the sticky piston again.[^3]

[^3]: Here's a brief explanation of the piston mechanics involved in the 1.3.2 BUD device, readers only need to understand. The sticky piston doesn't receive any NC Update when extending, so it won't drop the block midway. After the sticky piston extends into position, it triggers self-inspection, finds it's not powered, and executes retraction. During retraction, due to the block arrival order of sticky pistons, the piston arrives first. At this point, self-inspection is triggered but the redstone block hasn't arrived yet (in block entity state), and the redstone block arrives later. Therefore, the sticky piston won't extend again due to self-inspection.

## PP Update

### Concept and Behavior of PP Update

Almost all changes emit **PP Updates**. Except for:

- Placing blocks using commands
- Changing block states using the debug stick

and other extremely special situations; but there's still one exception:

- When a sticky piston pushes an activating observer, the observer doesn't emit a PP Update when it arrives.

> In the official deobfuscation, PP Update is `updateShape`, meaning update shape (for example, the shape of various walls, opening/closing of trapdoors, activation state changes of redstone components can all be called "shape" here). This deobfuscation name is more helpful for understanding PP Updates. The PP Update naming comes from the `postPlacement` method in mcp deobfuscation. In yarn deobfuscation it's called `getStateForNeighborUpdate`

Unlike NC Updates, the order of PP Updates is **West East North South Down Up**.

The behaviors mentioned above that *do not emit **NC Updates*** are all typical behaviors that emit **PP Updates** but not **NC Updates**. Similarly, the **BUD devices** mentioned above cannot detect **PP Updates**.

For example, here the fence gate state changes, emitting a PP Update, but it won't be detected by the piston BUD device

![Example](./img/01-budexample2.gif)

It should be noted that, except for placing and breaking blocks, the NC Update range and PP Update range of most redstone components are different. For example, the NC Update range of a repeater is the output end and the neighbors of the output end (except the repeater itself), while its PP Update range is the neighbors of itself.

PP Updates can be understood as updates for **changes to itself**, while NC Updates can be understood as updates to **notify blocks that may need to change state**.

Taking the repeater as an example, the repeater's on/off state is a **change to itself**, notifying adjacent blocks through PP Updates. When a repeater lights up, it powers the block at its output end. If the block at the output end can transmit redstone signals, this block will further transmit redstone signals to its neighbors; so the repeater lighting up may affect the block at its output end and the blocks adjacent to the output end. Therefore, the repeater uses NC Updates to **notify blocks that may be affected to check whether they should change state**.

Another example: when a dropper is activated and drops an item, the action of dropping an item does not affect surrounding blocks[^4], so dropper activation does not emit NC Updates. However, the dropper does change its activation state, and this state change causes the dropper to emit PP Updates.

[^4]: Not considering items being picked up by hoppers. Hopper pickup is the hopper's own behavior, not controlled by updates.

This way, the difference between PP Updates and NC Updates and their different update ranges become easier to understand.

### Properties of PP Update

Since the BUD devices mentioned above cannot detect PP Updates, what devices can detect (respond to) PP Updates?

The answer is—observers. The face of an observer that looks like a face is the side that detects PP Updates, and the side with a red dot is the side that outputs redstone signals.

Observers respond to, and only respond to, PP Updates.[^5] However, since in most cases NC Updates are always accompanied by PP Updates, the "only" here may not be very obvious. But we can still demonstrate this through some methods: when an observer directly faces a repeater, the repeater's on/off state will activate the observer; but if the observer faces the block (air) pointed to by the repeater's output end, the repeater's on/off state will not activate the observer. This is also easy to understand: when a repeater points to air, the air won't have any changes, so naturally the observer won't respond.

Through the combined use of BUD devices and observers, we can conveniently test the NC Update range and PP Update range of various redstone components without looking at the source code.

[^5]: Disambiguation: This is only considered from the perspective of update types. Observers only judge whether they should change and execute specific behaviors when receiving PP Updates. If "respond" is considered to mean "observer changes" in a broad sense, then besides the observer's scheduled tick behavior, when an observer is pushed into position by a piston, it calls its own `onBlockAdded` method. Only when there's no observer scheduled tick at the arrival position and the observer's original state was lit, the observer continues to call the `setBlockstate` method within it, with the final state becoming extinguished (no scheduled tick exists), while observers originally extinguished don't produce any changes. The scheduled tick added when an extinguished observer arrives via piston comes from TE's "update" `postProcessState` on the observer, which actually still calls the observer's PP response function `getStateForNeighborUpdate`.

## Comparator Update

### Concept and Behavior of Comparator Update

As the name suggests, Comparator Updates are updates specifically for updating comparators.

The following behaviors emit Comparator Updates:

- General containers, such as chests, barrels, hoppers, etc., when the number of items changes. Signal strength calculation method see below.
- Containers in a broader sense, blocks with a "capacity" concept such as composters (signal strength equals composting level, range 0\~9), cauldrons (signal strength equals liquid level, range 0\~3).
- Item frames. Item frames output comparator signal strength depending on the rotation angle of the displayed item, only outputting signals with strength 1\~8. When no item is placed, signal strength is 0.
- Lecterns. Signal strength calculation method see below.
- Detector rails with dropped items above them.
- When detector rails have minecarts with containers on them, such as minecart with chest or minecart with hopper, when the number of items in the minecart changes, a Comparator Update is emitted after 20gt. Signal strength calculation is the same as general containers.
- Chiseled bookshelves (1.20+). Signal strength equals the position of the last book placed or removed, first row is 1\~3, second row is 2\~6.

Comparators can both directly detect container capacity and detect container capacity through an intervening block that can transmit redstone signals. Therefore, the range of Comparator Updates is comparators within the horizontal adjacent range of the container, or comparators within the second-order adjacent range separated by a block that can transmit redstone signals. Comparator Updates do not update blocks other than comparators.

### Comparator Signal Strength Calculation

For general containers:

- Output signal strength = floor of `average fill ratio of each item slot * 14`. If the container is not empty, then output signal strength `+1`. That is: when the container is empty, output 0. When the container is not empty, output the mapping of the container's average fill ratio to 1~15 and floor it.

For lecterns:

- When no book is placed on the lectern, output signal strength is 0.
- When a book is placed on the lectern and the book has only one page, output signal strength is 15.
- When a book is placed on the lectern and the book has more than one page, output signal strength is floor of `current page number / total pages * 14` then `+1`.
- That is: when the lectern is empty, output 0. When the lectern is not empty, output the mapping of the current page number to total pages ratio to 1~15 and floor it.

### Comparator Update Detector (CUD)

When a comparator's current state differs from its **intended state**, this comparator constitutes a Comparator Update Detector (CUD). A simplest example is using a comparator to detect a filled composter through an intervening block, then using a piston to push the composter away. At this point, the comparator still has output signal.

Due to the complexity of other resettable CUD structures and their underlying principles, they are only demonstrated here, not expanded upon.

## Block Self-Inspection

Redstone components perform **self-inspection** when placed, i.e., checking whether they should change state. For example, place a redstone block, then place a repeater with its input end adjacent to the redstone block. At this point, the repeater hasn't received an update, but still lights up. This is because the repeater performed **self-inspection** when placed, found it should light up, and thus lit up.

A piston arriving at its position can also be viewed as a type of "being placed", so pistons also perform **self-inspection** once when arriving after extending and after retracting.

## [!ADVANCED] Invocation and Response of NC Update and PP Update

### `setBlockState` Function

`setBlockState` controls update behavior through a `bitflags` i.e. `FLAG`, in the `World` class in `World.java`:

```java
public boolean setBlockState(BlockPos pos, BlockState state, int flags, int maxUpdateDepth) {
        ...
        BlockState blockState = worldChunk.setBlockState(pos, state, (flags & Block.MOVED) != 0);
        if (blockState != null) {
            BlockState blockState2 = this.getBlockState(pos);
            if (blockState2 == state) {
                ...
                // NC Update invocation controlled by FLAG
                if ((flags & Block.NOTIFY_NEIGHBORS) != 0) {
                    this.updateNeighbors(pos, blockState.getBlock());
                    if (!this.isClient && state.hasComparatorOutput()) {
                        this.updateComparators(pos, block);
                    }
                }
                // PP Update invocation controlled by FLAG
                if ((flags & Block.FORCE_STATE) == 0 && maxUpdateDepth > 0) {
                    int i = flags & ~(Block.NOTIFY_NEIGHBORS | Block.SKIP_DROPS);
                    blockState.prepare(this, pos, i, maxUpdateDepth - 1);
                    state.updateNeighbors(this, pos, i, maxUpdateDepth - 1);
                    state.prepare(this, pos, i, maxUpdateDepth - 1);
                }
                ...
            }
            return true;
        }
        return false;
    }
```

### Invocation and Response of NC Update

In the source code, NC Updates are mainly invoked through two methods:

- `updateNeighbor` and its derivatives
- `setBlockState` with `flag` where `bit0 = 1`

`updateNeighbor` and its derivatives, i.e., `updateNeighbor` `updateNeighborsExcept` `updateNeighborsAlways`.

Among them, `updateNeighbor` is the most fundamental method, taking three parameters: `pos, sourceBlock, sourcePos`, which are the coordinates of the block to be updated `pos`, the block type emitting the update `sourceBlock`, and the coordinates of the block emitting the update `sourcePos`.

`updateNeighborsAlways` and `updateNeighborsExcept` perform updates by calling `NeighborUpdater.updateNeighbors`, in the `NeighborUpdater` instance in `NeighborUpdater.java`:

```java
public static final Direction[] UPDATE_ORDER = new Direction[]{Direction.WEST, Direction.EAST, Direction.DOWN, Direction.UP, Direction.NORTH, Direction.SOUTH};

default public void updateNeighbors(BlockPos pos, Block sourceBlock, @Nullable Direction except) {
    for (Direction direction : UPDATE_ORDER) {
        if (direction == except) continue;
        this.updateNeighbor(pos.offset(direction), sourceBlock, pos);
    }
}
```

Therefore, the NC Update order is West East Down Up North South.

Defined in `Block.java`:

```java
// Broadcast block update events to surrounding blocks
public static final int NOTIFY_NEIGHBORS = 1;

// Notify listeners and clients that need to react to this block change
public static final int NOTIFY_LISTENERS = 2;

// Default behavior when block changes: same effect as enabling both NOTIFY_NEIGHBORS and NOTIFY_LISTENERS above.
public static final int NOTIFY_ALL = 3;

// Skip updates, force set block state.
public static final int FORCE_STATE = 16;
```

For the above invocation methods, only the `sourcePOS` of calling `updateNeighborsAlways` and `updateNeighborsExcept` and the `pos` of `setBlockState` can be called an "update source", i.e., the "core" that updates the six adjacent blocks (or blocks excluding a certain direction) centered on one block. While `updateNeighbor` is a simple direct "update".

For PP Update only, `FLAG` is usually `NOTIFY_LISTENERS`; when emitting both PP Update and NC Update, `FLAG` is usually `NOTIFY_ALL`. Some blocks that override the `onBlockAdded` method have different invocation methods, see [below](#174-红石元件的nc更新与pp更新的调用)

In each block, NC Updates are responded to by the block's `neighborUpdate` function.

### Invocation of PP Update

In the source code, PP Updates are invoked through `setBlockState`. When `bit4 = 0` in `FLAG`, a PP Update is generated.

`state.updateNeighbors` is defined in the `AbstractBlockState` class in `AbstractBlock.java`:

```java
public final void updateNeighbors(WorldAccess world, BlockPos pos, int flags, int maxUpdateDepth) {
    BlockPos.Mutable mutable = new BlockPos.Mutable();
    for (Direction direction : DIRECTIONS) {
        mutable.set((Vec3i)pos, direction);
        world.replaceWithStateForNeighborUpdate(direction.getOpposite(), this.asBlockState(), mutable, pos, flags, maxUpdateDepth);
    }
}
```

In `AbstractBlock` there is:

```java
protected static final Direction[] DIRECTIONS = new Direction[]{Direction.WEST, Direction.EAST, Direction.NORTH, Direction.SOUTH, Direction.DOWN, Direction.UP};
```

Therefore, PP Update order is West East North South Down Up.

Where `world.replaceWithStateForNeighborUpdate`:

```java
default public void replaceWithStateForNeighborUpdate(Direction direction, BlockState neighborState, BlockPos pos, BlockPos neighborPos, int flags, int maxUpdateDepth) {
    NeighborUpdater.replaceWithStateForNeighborUpdate(this, direction, neighborState, pos, neighborPos, flags, maxUpdateDepth - 1);
}
```

Calls the `replaceWithStateForNeighborUpdate` method in `NeighborUpdater`:

```java
public static void replaceWithStateForNeighborUpdate(WorldAccess world, Direction direction, BlockState neighborState, BlockPos pos, BlockPos neighborPos, int flags, int maxUpdateDepth) {
    BlockState blockState = world.getBlockState(pos);
    BlockState blockState2 = blockState.getStateForNeighborUpdate(direction, neighborState, world, pos, neighborPos);
    Block.replace(blockState, blockState2, world, pos, flags, maxUpdateDepth);
}
```

Finally calls each block's `blockState`'s `getStateForNeighborUpdate` method. This is the method by which blocks respond to PP Updates.

### NC Update and PP Update Invocation for Redstone Components

For most redstone components, when calling `setBlockState` they usually pass `FLAG=2`. NC Updates are actually emitted by the `worldChunk.setBlockState` method calling the overridden `@Override` `onBlockAdded` method:

In `worldChunk.setBlockState`:

```java
public BlockState setBlockState(BlockPos pos, BlockState state, boolean moved) {
    ...
    if (!this.world.isClient) {
        state.onBlockAdded(this.world, pos, blockState, moved);
    }
    ...
    return blockState;
}
```

While PP Updates are normally invoked by `setBlockState` controlled by `FLAG`.

Taking the redstone repeater as an example, in the `AbstractRedstoneGateBlock` in `AbstractRedstoneGateBlock.java` inherited by the redstone repeater:

```java
public void scheduledTick(BlockState state, ServerWorld world, BlockPos pos, Random random) {
    if (this.isLocked(world, pos, state)) {
        return;
    }
    boolean bl = state.get(POWERED);
    boolean bl2 = this.hasPower(world, pos, state);
    if (bl && !bl2) {
        world.setBlockState(pos, (BlockState)state.with(POWERED, false), Block.NOTIFY_LISTENERS);
    } else if (!bl) {
        world.setBlockState(pos, (BlockState)state.with(POWERED, true), Block.NOTIFY_LISTENERS);
        if (!bl2) {
            world.scheduleBlockTick(pos, this, this.getUpdateDelayInternal(state), TickPriority.VERY_HIGH);
        }
    }
}


@Override
public void onBlockAdded(BlockState state, World world, BlockPos pos, BlockState oldState, boolean notify) {
    this.updateTarget(world, pos, state);
}


    protected void updateTarget(World world, BlockPos pos, BlockState state) {
    Direction direction = state.get(FACING);
    BlockPos blockPos = pos.offset(direction.getOpposite());
    world.updateNeighbor(blockPos, this, pos);
    world.updateNeighborsExcept(blockPos, this, direction);
}
```

`scheduledTick` is the method called when the repeater executes a scheduled tick. When the repeater executes a scheduled tick, it calls the `setBlockState` function, which then calls `worldChunk.setBlockState`, and in `worldChunk.setBlockState` the `@Override` `onBlockAdded` is called, generating NC Updates for the block pointed to by the repeater's output end and the neighbors of the pointed block (except the repeater itself). Then `setBlockState` calls `state.updateNeighbors` to generate PP Updates.

### Order of NC Update and PP Update

In summary, for most blocks that emit updates through `setBlockState`, they satisfy the order of NC Update first, PP Update second. This point is extremely important, as it affects the order issue of NC Updates and PP Updates in subsequent updates. This article does not expand on "continuous block updates".

### Observer's NC and PP Update

**The observer is an exception**. It doesn't call NC Updates in the `onBlockAdded` method, but directly calls `setBlockState` first in `scheduledTick` to emit PP Updates, then calls `updateNeighbors` to emit NC Updates. Therefore, observers follow the order of PP first then NC when lighting up and extinguishing.

In the observer's `onBlockAdded`:

```java
@Override
public void onBlockAdded(BlockState state, World world, BlockPos pos, BlockState oldState, boolean notify) {
    if (state.isOf(oldState.getBlock())) {
        return;
    }
    if (!world.isClient() && state.get(POWERED).booleanValue() && !world.getBlockTickScheduler().isQueued(pos, this)) {
        BlockState blockState = (BlockState)state.with(POWERED, false);
        world.setBlockState(pos, blockState, Block.NOTIFY_LISTENERS | Block.FORCE_STATE);
        this.updateNeighbors(world, pos, blockState);
    }
}
```

The first `if` contains `state.isOf(oldState.getBlock())` which is used to determine whether the block at the current position was still the current block before its state was changed. For example, if a piston pushes an observer, the position where the observer arrives is `air` before the change and `observer` after the change, at which point this function returns `true`. While a normal observer always returns `false` when calling this function.

Therefore, the observer's `onBlockAdded` method usually doesn't execute subsequent content, and thus doesn't emit NC Updates from `onBlockAdded`. Instead, as mentioned above, it "directly calls `setBlockState` first in `scheduledTick` to emit PP Updates, then calls `updateNeighbors` to emit NC Updates."

## [!ADVANCED] Control of Update Behavior

In the ServerWorld.setBlockState method, an integer parameter is passed in: flag
This flag is actually treated as a 9-bit binary code, where each of the 9 bits controls different behaviors

## Definition of flag's 9-bit Flags

- **Bit 0**: Emit neighbor updates.
- **Bit 1**: Notify listeners (server and client).
- **Bit 2**: Suppress client rendering.
- **Bit 3**: Force client sync redraw.
- **Bit 4**: Force state storage.
- **Bit 5**: Prevent item drops.
- **Bit 6**: Mark piston movement.
- **Bit 7**: Light update.
- **Bit 8**: Trigger behavior side effects.

## Constant Explanation

Multiple update behaviors are defined in Block.java, they are:

|--------------------------------------------------|-----|----------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------|--------------------------------|
