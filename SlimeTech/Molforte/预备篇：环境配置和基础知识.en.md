#### Necessary Notes

- Structure names cannot use Chinese characters.
- Relative position and structure size both have three boxes. These three boxes from left to right correspond to the x-axis, y-axis, and z-axis, which can be referenced by pressing "F3" on the keyboard and looking at the crosshair in the center of the screen.
- **Save mode can only save structures with 0° angle in the corresponding load mode structure box**.
- Include entities: In save mode, saves entities within the structure box; in load mode, loads entities that were saved in save mode.

### Carpet Commands

Below is a brief description of Carpet configuration. For more options, please refer to other chapters or watch Carpet usage tutorials.

#### Carpet Configuration

Open your test world and enter commands in the chat box. Once set in this subsection, these settings generally won't need to be changed again.

`/carpet tntDoNotUpdate true`: When placing TNT, it won't be activated by redstone. This saves us from pushing minecarts or other "loading" methods when making TNT duplicators.

`/carpet fillUpdates false`: Using fill, clone, setblock commands and structure blocks won't cause updates. This especially solves the problem of machines "moving randomly" after loading structures.

`/carpet creativeNoClip true`: No collision in creative mode, allowing us to happily shuttle through various machine structures while solving the problem of being bounced away by slime blocks.

`/carpet structureBlockLimit 48/96/192/256`: The numeric options are separated by "/". When you need a structure range larger than the default selection range of structure blocks, consider this option. **Note that in actual use, overly large structure boxes can cause rendering issues and lag. Unless necessary, the default option of 48 is sufficient**.

`/carpet explosionNoBlockDamage true`: When designing structures related to TNT, you can turn on this debug option, which makes TNT explosions not destroy any blocks; when you need to test whether TNT will destroy the expected range, you can set it to false.

#### Game Tick Commands

- `/tick freeze`: Freezes game ticks, usually combined with the `/tick step x` command below to observe whether the designed structure operates as expected. Similar to debugging breakpoints in programming.
- `/tick step x`: After freezing game ticks, this command advances the game by x game ticks, where x is the number of game ticks to advance.
- `/tick sprint x`: Similar to `/tick warp`, usually used for long-term machine stability testing or accelerating flying machine positioning (for debugging convenience).
- `/tick unfreeze`: Unfreezes game ticks. Some versions use `/tick freeze` to toggle freezing game ticks on and off. <hidden>Complaining a bit, I don't know why they split one command into two; typing commands is so troublesome every time.</hidden>

#### TNT Monitoring Commands

- `/log tnt`: Use this command to subscribe to TNT-related options.
- `/log tnt full` and `/log tnt brief`: When TNT explodes, tells you the TNT spawn position and explosion point position, very useful when controlling TNT explosion points. We'll cover this in the practical section.
- `/log tnt clear`: Unsubscribe from TNT-related options.

#### Fake Player Commands

- `/player <Name> spawn`: The first step in using fake players; the most basic use is to force-load chunks when testing machines. For example, use `/player yu_xian520 spawn` to summon a fake player named Yu Xian. <hidden>Follow Sanri Yuyin on Bilibili meow~</hidden>
- `/player <name> use`: Makes the fake player use items, place blocks, or interact with blocks, equivalent to right-clicking. A similar command is `/player <name> use once`. Aiming the fake player at a note block/fence gate can achieve remote startup/stepping effects. For example, using `/player Twisuki use` makes Twisuki click a note block/fence gate once.
- `player <name> use continous`: Makes the fake player continuously use items, place blocks, or interact with blocks. Usually used to place pistons for bedrock breakers. Use `/player <name> use` to end placement.

### WorldEdit Commands

_This section mainly covers the use of `//set`, `//undo`, `//redo`, `//stack`, `//move` and other commands. Players who have already mastered these can skip to the next section._
**_This subsection only covers the most commonly used parts in slime tech design_**.

1. //set, //undo, //redo
   //set is a very basic command, right? Using the wooden axe, like with the projection mod, left and right click to select a certain range, then enter `//set 0`

If nothing goes wrong, the area you selected has been replaced with air. So, how do you restore this platform?

- Use `//undo` to undo this operation;
- Use `//redo` to redo the operation you just undid.

1. `//move <Num>`
   This is also a very basic command. After selecting an area, face a certain direction (or use `//set <Num> <direction>` to move the blocks in that area. _For example, use //move 1 up to move up one block_. Without specifying a direction, it defaults to the player's facing direction. Use `//move <Num> -e` to preserve entities.

You can use this command to move created modules to appropriate positions.

1. `//stack <Num>`
   After selecting an area, you can batch copy the area in a certain direction, very convenient for expanding slime tech units.
   This command usage is basically similar to `//move`, except it changes from "move how many blocks" to "copy how many times". No need to elaborate further.

### Tweakeroo Configuration

1. **Disable Observer Option**:
   In the game interface, press `x+c` simultaneously. In the disable options, you can find the disable observer option. I suggest setting a shortcut key on the right side of the keyboard for easy access when debugging.

After enabling this option, observers become "stone" and no longer respond to updates. When we use WE or other tools to operate on slime tech structures, this avoids unpredictable results from observer activation. It also prevents us from accidentally updating observers.

1. **Flyspeed Flight Speed Control Flight Speed Preset**:
   Tunnel borers, mining machines, etc. often have heights exceeding 200 blocks. Moving around various parts of the machine is very troublesome. Using the Flyspeed function can greatly increase the player's flight speed, allowing quick travel between different parts.

### Pistorder Usage

This is one of the essential mods for designing slime tech. Previous chapters have also covered pistorder and piston push order related rules. Here's just a brief description of slime tech-related aspects.

Right-click a regular piston with an empty hand, and the piston will display whether it can push (as shown in the image: Push √; if it cannot push, it displays Push ×). The blocks being pushed display their push order **from small to large**. This helps us compress machines and debug quickly.

- For regular pistons: Right-click once with empty hand for direct push (piston arm directly contacts), right-click twice for pushing with one block gap (as shown below).
- For sticky pistons: Right-click once with empty hand for direct push, right-click twice for pulling back with one block gap.

## Fundamentals

### Timing Basics

#### Game Tick

We've already learned the basic content of game ticks in previous chapters. This section is more inclined to introduce the application of game ticks in slime tech.

To accommodate readers who haven't studied the timing chapter, here's a brief summary of the more important basic content:

- The basic unit of time in Minecraft is the game tick (abbreviated as **gt**).
- Different redstone components have different game tick delays. For example, observers have a 2gt delay, and repeaters delay 2gt per notch (so repeaters delay 2~8gt).

All content in this chapter will abandon the use of **Redstone Tick (RedTick, abbreviated as rt, 1rt = 2gt)**, because we will later involve certain odd-tick timing relationships, and using game ticks is more convenient for calculation.

For slime tech, game ticks can be used to indicate the sequential movement relationship of each wiring. For most cases, we only need to know their movement order. Below is my recreation of my first slime tech machine, a tunnel boring machine using piston chains as delays.

**Image: Piston delay-type tunnel boring machine**

_Readers can also remember that the upper sticky piston section has a 6gt delay, and the lower regular piston has a 4gt delay_.

Players familiar with slime tech know that signal transmission starts from the upper sticky piston chain and ends at the lower regular piston chain. _This structure is used as TNT delay for tunnel boring machines, etc._ After becoming familiar with the signal delay above, we can calculate that the total delay in the above image is the sum of multiple sticky pistons and regular pistons, i.e., the sum of multiple 6gt and multiple 4gt.

Of course, through simple calculation we also know that with the above piston delay arrangements and combinations, we can create delays of 4gt and above, all even-numbered gt delays. **So, how do we create a 2gt delay?** This seems a bit troublesome. We can usually only change the 6gt delay to a 4gt delay to advance by 2gt, or vice versa.

Since we already have even-numbered gt delays, then, **how do we create odd-numbered gt delays?** This seemingly difficult task is actually much simpler! Let's return to the 4gt delay piston structure in the image above. We can transform it into a transmission structure based on redstone blocks. With the redstone block relationship, the piston will trigger the next gt after arriving, thus creating a 3gt signal delay.

**Image: Diagram of two types of piston delays (puzzle, combine two images into one)**

_Let's analyze the specific situation: For the observer structure in the upper image, at gt0 the piston and observer arrive, at gt1 the observer activates, at gt2 the piston extends, at gt3 the piston is extending, at gt4 the pushed block arrives; for the redstone block structure in the lower image, at gt0 the piston arrives, at gt1 the piston extends, at gt2 the piston is extending, at gt3 the pushed block arrives_.

Combining the above 4gt and 6gt delay units, we can create 3gt and 7gt and above delay units.

Below we discuss a wonderful 7gt delay technique: Let's return to the 6gt delay structure in the image above. As shown, we place a powered block diagonally above and add another observer to QC it, creating a simple and clever 7gt delay. This is also applied in kades' three-way bedrock breaker:

**Image: 7gt delay structure, with application part from kades' bedrock breaker**.

Of course, based on the above principle, we can create flying machines with an 11gt cycle, or 11gt two-step structures, which are the finishing touch when applied to trenchless world eaters.

Of course, in many cases, we don't need to calculate delays so precisely. Many slime tech structures only need to reset before the next activation. Precisely because many structures are so-called "insensitive," there is a rather clever wiring style: **mounting/borrowing force**.

#### Mounting/Borrowing Force

Since I first got into slime tech, I've heard something like this:

> Excellent wiring often applies a lot of mounting/borrowing force.

As mentioned in the previous section, mounting/borrowing force is a wiring technique that resets structure A, which is not time-sensitive, through other structure B or even completely unrelated structure C, or even directly replaces structure A's wiring. Very convoluted, right? Let's look directly at the image:

**Image: Mounting/borrowing force diagram, preferably GIF**

As shown in the image, the upper piston with no-delay linkage, after triggering, has its sticky piston reset by a lower no-delay linkage that may serve other purposes. This is mounting/borrowing force. This can serve as a relatively simple example.

Combined with the game tick section above, we can use a certain moment of machine operation as the time origin (similar to the zero reference point of potential in circuit analysis. <hidden>Students in electronics are no strangers to this</hidden>). Each module and even each piston can get a gt number label, so we can accurately analyze the sequential movement order of each part.

**Image: My 14m high front-loading trenchless world eater**

As shown, this is the departure station part of my 14m front-loading trenchless world eater. Using the signal output part of the TNT buffer as the 0gt point, we can measure that the time interval for launching the flying machine is 7gt. (Of course, different marking points will yield different times).

Except for situations with extreme requirements on machine size or timing, most of the time there's no need to calculate timing so rationally. This subsection only provides a rational analysis approach to meet needs in extreme situations.
